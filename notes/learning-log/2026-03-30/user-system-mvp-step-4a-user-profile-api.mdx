## Context
Implemented the first `/api/user/*` endpoint for authenticated profile reads/updates.

## Goal
Add `GET` and `PUT` support for user profile data with request validation and auth checks.

## Files Changed
- `app/api/user/profile/route.ts`

## Implementation Notes (key logic explained)
- Added auth gate using `getAuthUserFromRequest`.
- `GET` behavior:
  - fetch profile by `auth.userId`
  - auto-bootstrap profile row if missing
- `PUT` behavior:
  - validates request body with zod
  - upserts profile fields (`username`, `displayName`, `avatarUrl`, `timezone`)
- Returns camelCase JSON payload while storing snake_case DB columns.

## Key Decisions (and alternatives considered)
- Decision: bootstrap profile row lazily on first GET.
  - Alternative: create profile only via explicit onboarding endpoint; skipped to reduce friction.
- Decision: allow nullable profile fields.
  - Alternative: force required display name/username at creation; skipped for smoother MVP activation.

## Assumptions
- RLS policies are active and user id from auth token maps to `profiles.id`.
- Client sends valid auth credentials via header or Supabase cookie session.

## Failure Modes
- Missing/invalid auth token returns 401.
- Missing Supabase env vars in auth helper path returns 500 from thrown config error.
- Username uniqueness collisions propagate from DB as save failure.

## How to Test (commands + steps)
1. Call `GET /api/user/profile` unauthenticated; expect 401.
2. Call authenticated `GET`; expect profile object (auto-created if first time).
3. Call authenticated `PUT` with valid fields; expect updated profile.
4. Call `PUT` with invalid `avatarUrl`; expect 400 validation error.
5. Confirm DB row updates in `public.profiles`.
