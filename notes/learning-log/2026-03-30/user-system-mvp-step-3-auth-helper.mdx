## Context
Added server-side auth context resolution for new user APIs.

## Goal
Create a reusable helper that resolves the authenticated Supabase user from incoming Next.js API requests.

## Files Changed
- `lib/auth/supabase-user.ts`

## Implementation Notes (key logic explained)
- Added `getAuthUserFromRequest(request)` to:
  - read bearer token from `Authorization` header, or
  - read access token from Supabase cookie patterns (`sb-access-token` or `*-auth-token`).
- Validates token by calling Supabase endpoint: `GET /auth/v1/user`.
- Returns typed auth context: `userId`, `email`, `accessToken`; returns `null` when unauthenticated.

## Key Decisions (and alternatives considered)
- Decision: use direct `fetch` to Supabase Auth API instead of SDK dependency.
  - Alternative: install `@supabase/supabase-js`; skipped for smaller immediate diff.
- Decision: support both header and cookie token sources.
  - Alternative: require header only; skipped to support browser session flows.

## Assumptions
- Environment includes Supabase URL + anon key:
  - `NEXT_PUBLIC_SUPABASE_URL` or `SUPABASE_URL`
  - `NEXT_PUBLIC_SUPABASE_ANON_KEY` or `SUPABASE_ANON_KEY`

## Failure Modes
- Missing env vars causes explicit error throw in auth helper.
- Invalid/expired tokens produce `null` auth context.
- Non-standard cookie shape may require parser updates.

## How to Test (commands + steps)
1. Set required Supabase env vars locally.
2. Call an API route using this helper without auth; expect 401.
3. Call with valid bearer token; expect authenticated response.
4. Call from browser with Supabase cookie session; expect authenticated response.
5. Call with expired token; expect 401.
