## Context
Connected props explorer UI to user saved-props backend for auth-aware bookmarking.

## Goal
Add Save/Unsave UX in props explorer rows and sync UI state with `/api/user/saved-props`.

## Files Changed
- `app/betting/props-explorer/page.tsx`

## Implementation Notes (key logic explained)
- Added saved props state map: `savedPropIdByKey`.
- Added stable key builder (`gameId|playerId|sportsbook|propType|side|lineValue|snapshotAt`) used for row/save matching.
- On mount:
  - fetches `/api/user/saved-props?limit=200`
  - handles 401 by clearing saved state silently
- Added row-level save toggle:
  - if already saved, calls `DELETE /api/user/saved-props?id=...`
  - otherwise calls `POST /api/user/saved-props` with row payload
- Added Save column and button state (`Save` / `Saved` / loading dots).

## Key Decisions (and alternatives considered)
- Decision: keep save state in local component map keyed by prop identity.
  - Alternative: refetch saved-props after each toggle; skipped for lower latency/UI flicker.
- Decision: do not force redirect on 401.
  - Alternative: route to sign-in page; skipped because auth UI flow is not part of this MVP task.

## Assumptions
- Saved-props API is available and authenticated by cookie/bearer context.
- Current row fields in props explorer are sufficient for saved-prop identity.

## Failure Modes
- If auth is missing, save actions return 401 and surface API error message.
- If API returns malformed payload, saved state map may not update.
- With very large saved sets (>200), initial saved-state sync may be incomplete.

## How to Test (commands + steps)
1. Open props explorer while signed out; Save buttons should render but API calls should fail with auth error.
2. Sign in and refresh; previously saved rows should show `Saved`.
3. Click `Save` on a row and verify it becomes `Saved`.
4. Click `Saved` on same row and verify it reverts to `Save`.
5. Confirm DB inserts/deletes in `public.user_saved_props`.
