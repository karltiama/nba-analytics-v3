## Context
`raw.player_prop_snapshots_v2` was growing rapidly because every lambda pull inserted new rows with no dedupe guard.

## Goal
Reduce raw v2 growth while preserving Prop Explorer reads from latest analytics tables.

## Files Changed
- `lambda/player-props-snapshot/src/bulk-writers.ts`

## Implementation Notes (key logic explained)
- Added `ON CONFLICT ... DO NOTHING` to `bulkInsertRawV2`.
- Conflict target now includes an hourly time bucket expression: `date_trunc('hour', fetched_at)`.
- This keeps at most one raw row per `(game_id, player_id, sportsbook, prop_type, side, line_value, hour)`.

## Key Decisions (and alternatives considered)
- Chosen: write-time dedupe in the insert statement.
- Alternative: post-write compaction jobs only. Rejected because it still allows uncontrolled short-term growth.

## Assumptions
- A matching unique index/constraint for the hourly conflict target will be added in migration (next step).
- Keeping one row per hour is sufficient for historical explorer constraints.

## Failure Modes
- If the unique index is missing, Postgres will error when using conflict inference.
- If `line_value` has precision drift, near-identical lines may still store multiple rows.

## How to Test (commands + steps)
1. Deploy migration adding hourly unique index.
2. Run the worker twice within the same hour for the same games.
3. Verify second run inserts fewer/no duplicate raw rows:
   - `select count(*) from raw.player_prop_snapshots_v2 where fetched_at >= now() - interval '1 hour';`
4. Verify `analytics.player_props_current` still updates normally.

## Next Steps
- Add migration for hourly uniqueness/index support.
- Add retention pruning and optional `raw_json` storage controls.
