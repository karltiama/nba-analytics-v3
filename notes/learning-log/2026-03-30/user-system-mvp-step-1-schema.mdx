## Context
Started implementing the Supabase user system MVP with core user data tables.

## Goal
Create the foundational database schema for user identity, user preferences, and saved props.

## Files Changed
- `db/schemas/MIGRATION_user_system_mvp.sql`

## Implementation Notes (key logic explained)
- Added `public.profiles` keyed by `auth.users(id)` for Supabase-authenticated identity.
- Added `public.user_settings` as a 1:1 table keyed by `user_id`.
- Added `public.user_saved_props` for bookmarking props tied to a user.
- Added indexes for lookup speed and a composite unique index to dedupe saved props.

## Key Decisions (and alternatives considered)
- Decision: use `uuid` PK on `user_saved_props` with `gen_random_uuid()`.
  - Alternative: `bigserial` PK; skipped for consistency with auth UUID usage.
- Decision: keep saved prop fields denormalized.
  - Alternative: reference-only pointer model; skipped to keep read path simple for MVP.

## Assumptions
- Supabase auth schema (`auth.users`) is available in the target database.
- Saved props are read mostly by `user_id` and newest-first.

## Failure Modes
- If `auth.users` is unavailable in non-Supabase environments, migration will fail.
- Composite dedupe key may need refinement if product later allows multiple snapshots for same prop identity.

## How to Test (commands + steps)
1. Apply migration in SQL editor or migration runner.
2. Verify tables exist: `profiles`, `user_settings`, `user_saved_props`.
3. Insert a test profile row using an existing auth user id.
4. Insert two identical saved props and confirm unique index blocks duplicates.
5. Run indexed query: fetch saved props by `user_id` ordered by `created_at desc`.
