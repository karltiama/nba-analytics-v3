## Context
Extended the user-system migration with ownership enforcement and timestamp hygiene.

## Goal
Ensure only authenticated owners can access/modify their rows, and keep `updated_at` fields current on updates.

## Files Changed
- `db/schemas/MIGRATION_user_system_mvp.sql`

## Implementation Notes (key logic explained)
- Added shared trigger function `public.set_updated_at()` and attached it to:
  - `public.profiles`
  - `public.user_settings`
  - `public.user_saved_props`
- Enabled RLS on all three tables.
- Added owner-only policies for `select`, `insert`, `update`, and `delete`.
  - `profiles`: `id = auth.uid()`
  - settings/saved props: `user_id = auth.uid()`

## Key Decisions (and alternatives considered)
- Decision: explicit per-table, per-action policies for readability and auditing.
  - Alternative: fewer broader policies; skipped to keep intent obvious.
- Decision: one reusable `updated_at` trigger function.
  - Alternative: separate trigger functions per table; skipped as redundant.

## Assumptions
- API access to these tables will use authenticated Supabase JWT context.
- Service role usage is limited to trusted backend workflows.

## Failure Modes
- If requests are unauthenticated, table reads/writes will fail under RLS as expected.
- If backend bypasses auth context unintentionally, requests may see permission errors.

## How to Test (commands + steps)
1. Re-run migration SQL in Supabase.
2. As user A, insert/select/update own profile/settings/saved props.
3. As user B, attempt to read user A rows and confirm access denied/empty.
4. Update a row and verify `updated_at` changes.
5. Verify unauthenticated queries to these tables fail.
