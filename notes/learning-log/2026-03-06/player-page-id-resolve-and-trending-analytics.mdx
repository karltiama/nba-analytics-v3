# Player page ID resolve + trending from analytics

## Context

Player pages were refactored to use only the analytics schema (`analytics.players`, `player_game_logs`, `player_season_averages`). Entry points (e.g. betting dashboard trending, team roster) still used legacy player IDs (e.g. from bbref/public). The player page looked up only by `analytics.players.player_id` (BallDontLie IDs), so URLs with old IDs showed "Player not found."

## Goal

1. Resolve incoming URL `playerId` to analytics `player_id` so both old and new IDs work when possible.
2. Switch trending API to analytics so dashboard links use analytics player IDs and the player page works from the dashboard.

## Files Changed

- `lib/players/analytics-queries.ts` – Added `resolveAnalyticsPlayerId(playerId)`.
- `app/betting/players/[playerId]/page.tsx` – Call resolver before loading; if null, show not found.
- `lib/betting/queries.ts` – Added `getTrendingPlayersFromAnalytics(limit)` and `CURRENT_ANALYTICS_SEASON`.
- `app/api/betting/players/trending/route.ts` – Use `getTrendingPlayersFromAnalytics`, return `dataSource: 'analytics'`.

## Implementation Notes

- **resolveAnalyticsPlayerId:** (1) If `analytics.players` has a row with `player_id = playerId`, return it. (2) Else query `provider_id_map` with `entity_type = 'player'`, `provider = 'balldontlie'`, `internal_id = playerId`; if a row exists, return `provider_id` (analytics/BallDontLie id). (3) Else return null so the page can show "Player not found."
- **Page:** `loadPlayerAnalysis` now calls `resolveAnalyticsPlayerId(playerId)` first; if null, returns empty data (not found). Otherwise uses the resolved id for `getAnalyticsPlayerInfo`, `getAnalyticsPlayerSeasonStats`, `getAnalyticsPlayerGames`.
- **getTrendingPlayersFromAnalytics:** Uses `analytics.player_season_averages` (filter `season = '2025'`, `games_played >= 5`, `pts_avg >= 10`), L5 from `analytics.player_game_logs` + `analytics.games` (status = 'Final'), and `player_team` CTE (DISTINCT ON player from `player_game_logs` by latest `game_date`) to get current team. Returns same `TrendingPlayer[]` shape with `player_id` = analytics id. Trending API response `id` is that value, so dashboard links go to `/betting/players/{analytics_id}`.

## Key Decisions

- **Resolver on the page:** Keeps entry points unchanged except trending; bookmarks or other links with old IDs still work if `provider_id_map` is populated.
- **Trending from analytics:** Dashboard only shows players present in analytics and links with analytics id, so no mapping table is required for that flow. Season is hardcoded `'2025'`; can be made configurable later.
- **provider_id_map:** Used only when the URL id is not already an analytics id. If the map has no player rows, old-ID links will continue to show not found until the map is backfilled (e.g. script mapping legacy id → balldontlie id).

## Assumptions

- `provider_id_map` lives in `public` (or is in search path). Columns: `entity_type`, `provider`, `provider_id`, `internal_id`; for players we use `internal_id` = legacy id, `provider_id` = analytics/BallDontLie id.
- Analytics data is seeded and transformed for the current season (e.g. 2025); otherwise trending may return few or no players.

## Failure Modes

- Empty trending: no rows in `player_season_averages` for season 2025, or L5/join filters drop everyone. Check analytics seed + transform + compute-season-averages.
- Old ID still not found: `provider_id_map` has no row for that player. Add a backfill script or switch the source of that link to analytics id.
- Wrong team on trending: `player_team` uses latest game by `game_date`; trades mid-season can make "current" team wrong until more games are logged.

## How to Test

1. **Analytics ID in URL:** Open `/betting/players/<analytics_player_id>` (e.g. from API or DB). Page should load.
2. **Dashboard:** Open `/betting`, ensure trending cards load; click a player. URL should be `/betting/players/<analytics_id>` and page should load.
3. **Old ID:** If you have a legacy id and a row in `provider_id_map` (internal_id = legacy, provider_id = analytics), open `/betting/players/<legacy_id>` and confirm it loads.
4. **Not found:** Open `/betting/players/99999999` (no such analytics id, no map). Page should show not found.

## Next Steps

- Backfill `provider_id_map` for players (legacy id → balldontlie id) if other entry points (e.g. team roster) still use legacy ids.
- Make `CURRENT_ANALYTICS_SEASON` configurable (env or derived from date).
- Optionally add team roster/team page to use analytics player ids so all player links are consistent.
