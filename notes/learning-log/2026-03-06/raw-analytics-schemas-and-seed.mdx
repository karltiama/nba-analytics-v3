# Raw + Analytics Schemas and BallDontLie Seed Script

## Context

We want new data pipelines separate from existing public tables (teams, players, games, etc.). Later the old data will be removed. BallDontLie is used for teams, players, games, player game stats, and season averages.

## Goal

- Add two PostgreSQL schemas: **raw** (mirrors BallDontLie API) and **analytics** (normalized for the app).
- Add a seed script that fetches from BallDontLie and populates **raw** only. Transform from raw → analytics will be implemented later.

## Files Changed

- **Added:** `db/schemas/raw_schema.sql` — creates `raw` schema and tables: `raw.teams`, `raw.players`, `raw.games`, `raw.player_game_stats`, `raw.season_averages` with columns matching BallDontLie API (ids as integer, nested objects as jsonb where useful).
- **Added:** `db/schemas/analytics_schema.sql` — creates `analytics` schema and tables: `analytics.teams`, `analytics.players`, `analytics.games`, `analytics.player_game_logs` with normalized columns and FKs within analytics only.
- **Added:** `scripts/seed-raw-balldontlie.ts` — fetches teams, players, games from BallDontLie; optionally `--stats` and `--season-averages`; writes only to `raw.*`.

## Implementation Notes

- **Raw schema:** No foreign keys to other schemas. Integer IDs and jsonb for nested API objects (e.g. `home_team`, `visitor_team` on games). Denormalized `player_id`, `team_id`, `game_id` on `raw.player_game_stats` for easier querying; full `player`, `team`, `game` jsonb kept for fidelity.
- **Analytics schema:** Text primary keys (`team_id`, `player_id`, `game_id`), FKs only within analytics. `analytics.player_game_logs` uses app-friendly column names (e.g. `points`, `rebounds`, `personal_fouls`) and references `analytics.games`, `analytics.players`, `analytics.teams`. Not populated by the seed script.
- **Seed script:** Uses `SUPABASE_DB_URL` and `BALLDONTLIE_API_KEY`. Rate limit delay via `BALLDONTLIE_REQUEST_DELAY_MS` (default 12s). Order: teams → players → games; then optionally stats (by game_ids in batches of 25), then season_averages (one request per player per season). All endpoints use `https://api.balldontlie.io/v1` (teams, games, players, stats, season_averages).

## Key Decisions

- **Separate schemas:** Keeps raw API shape and normalized app shape isolated; existing public tables untouched until we delete old data.
- **Raw mirroring API:** Storing nested objects as jsonb preserves full API response for debugging and future schema changes without re-fetching.
- **Seed only raw:** Transform raw → analytics is a separate step (later) so we can change transform logic without touching the fetch step.
- **Optional stats and season_averages:** `--stats` and `--season-averages` flags to avoid long runs and rate limits when only schedule/rosters are needed.

## Assumptions

- BallDontLie API uses same base path `/v1` for teams, games, players, stats, season_averages (some docs show `/nba/v1`; script uses `/v1` which is documented for players).
- Existing public schema and tables are not modified; no migrations run against `teams`, `players`, `games` in public.

## Failure Modes

- If `/v1/players` or `/v1/stats` or `/v1/season_averages` return 404, BallDontLie may require `/nba/v1/` prefix for those; switch `BDL_BASE` to include path per endpoint if needed.
- Rate limits (429): script uses exponential backoff and per-request delay; for large seasons, `--stats` and `--season-averages` can hit limits.
- Raw tables must exist before running the seed (run `raw_schema.sql` first).

## How to Test

1. **Create schemas and tables (run in DB):**
   - `db/schemas/raw_schema.sql`
   - `db/schemas/analytics_schema.sql`
2. **Seed raw only:**
   - `npx tsx scripts/seed-raw-balldontlie.ts`
   - Or with date/season: `npx tsx scripts/seed-raw-balldontlie.ts --season 2025 --start 2025-10-01 --end 2025-11-01`
   - Optional: `npx tsx scripts/seed-raw-balldontlie.ts --stats --season-averages --season 2025`
3. **Verify:** `SELECT count(*) FROM raw.teams;`, `FROM raw.players;`, `FROM raw.games;`. Analytics tables remain empty until transform is built.

## Next Steps

- Implement transform: raw → analytics (map integer IDs to text, normalize column names, fill `analytics.teams`, `analytics.players`, `analytics.games`, `analytics.player_game_logs`).
- When ready, remove or migrate away from existing public tables and point app at `analytics.*`.
