# Fix legacy player ID resolution

## Context

Player pages showed "Player not found" because the dashboard linked to legacy `public.players` IDs (e.g. `1630595` for Cade Cunningham) while the player page only queried `analytics.players` which uses BallDontLie IDs (e.g. `17896075` for Cade). The two ID systems are completely different numbers for the same player.

## Goal

Make player pages work with both legacy IDs (from existing links/bookmarks) and analytics IDs (from the new trending API).

## Files Changed

- `lib/players/analytics-queries.ts` – Added name-based fallback to `resolveAnalyticsPlayerId`
- `scripts/backfill-provider-id-map-players.ts` – New script to populate `provider_id_map` with balldontlie player mappings

## Implementation Notes

**Resolver now has 3 resolution steps:**
1. Direct `analytics.players` lookup by `player_id` (handles analytics IDs)
2. Name match: `public.players.player_id` -> `full_name` -> `analytics.players.player_id` (handles legacy IDs)
3. `provider_id_map` fallback with `provider='balldontlie'` (for any future mapped IDs)

**Backfill script:** Joins `public.players` and `analytics.players` on `full_name`, inserts into `provider_id_map` with `entity_type='player'`, `provider='balldontlie'`, `internal_id=public_id`, `provider_id=analytics_id`. Idempotent via `ON CONFLICT DO NOTHING`. Inserted 508 out of 538 public players (30 have no analytics match, likely inactive/historic players).

## Key Decisions

- **Name-based match over map-only:** The resolver does a live JOIN on `full_name` as step 2, rather than relying solely on `provider_id_map`. This works immediately for any legacy ID without needing the backfill first.
- **Backfill as belt-and-suspenders:** The map is populated anyway so step 3 works too, and for potential future use by other systems.

## Assumptions

- `public.players.full_name` matches `analytics.players.full_name` for the same person (both come from the same name sources).
- Name collisions are rare enough that `LIMIT 1` in the name match is acceptable. If two players share a name, only one is matched.

## Failure Modes

- **Duplicate full names:** Two different players with the same name could cause the wrong player to load. Very rare in practice.
- **Player not in public.players:** If a link uses an ID that isn't in `public.players` at all (e.g. from a different system), all 3 resolution steps fail and "not found" is shown.

## How to Test

1. Open `/betting/players/1630595` (Cade Cunningham legacy ID) – should now load his profile
2. Open `/betting/players/17896075` (Cade Cunningham analytics ID) – should also load
3. Open `/betting/players/99999999` (invalid) – should show "Player not found"
4. Open `/betting` dashboard, click a trending player card – should link to analytics ID and load

## Next Steps

- The trending API now returns analytics IDs, so going forward dashboard links will use analytics IDs directly (no resolver fallback needed).
- Other entry points (team roster pages) may still use legacy IDs; those will now resolve correctly via the name match.
