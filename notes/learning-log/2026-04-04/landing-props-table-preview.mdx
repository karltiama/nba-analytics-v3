## Context

Landing page (`app/page.tsx`) showcased hero, featured games, and a static dashboard image. The Props Explorer table is a core product surface; we wanted a faithful visual preview without wiring live API data.

## Goal

Show a Props Explorer‚Äìstyle table on the landing page with a few illustrative rows and skeleton rows at the bottom to suggest ‚Äúmore data loading.‚Äù

## Files Changed

- `components/landing/LandingPropsTablePreview.tsx` ‚Äî new section: table shell matching explorer columns, static `DEMO_ROWS`, tail rows via `Skeleton` (pattern aligned with `PropsExplorerTableSkeleton`).
- `app/page.tsx` ‚Äî render `<LandingPropsTablePreview />` after `FeaturedGames`.

## Implementation Notes (key logic explained)

- **Structure**: Same `glass-card`, sticky header, and column set as `app/betting/props-explorer/page.tsx` (Player through Paper) so the preview reads as the same UI.
- **Data**: Hard-coded demo rows only; no `fetch`, no env vars, no new dependencies.
- **Tail**: `SKELETON_TAIL_ROWS = 3` reuses the same skeleton cell layout as `PropsExplorerTableSkeleton` for visual consistency; `aria-hidden` on skeleton rows since they are decorative.
- **Actions**: ‚ÄúSave‚Äù / ‚ÄúAdd‚Äù are non-interactive spans with muted styling so visitors are not misled into thinking props are savable from the marketing page; link ‚ÄúOpen live board‚Äù goes to `/betting/props-explorer`.

## Key Decisions (and alternatives considered)

- **Static vs API**: Chose static demo data to avoid auth, rate limits, and empty states on the public homepage. Alternative: server component fetching public snapshot endpoint ‚Äî heavier and needs API design.
- **Reuse `PropsExplorerTableSkeleton`**: Could import and render only tail rows, but that would require exposing a row component or row count prop; a local `SkeletonRow` keeps the landing module self-contained.

## Assumptions

- Demo numbers and names are illustrative only, not live odds.
- Three skeleton rows counts as ‚Äúlast few‚Äù for the teaser.

## Failure Modes

- **Narrow viewports**: Table is wide; horizontal scroll is required (`overflow-x-auto`) ‚Äî same as explorer.
- **Screen readers**: Decorative skeleton rows are `aria-hidden`; the section has a proper heading id.
