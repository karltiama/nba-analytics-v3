## Context

The betting app needed an MVP flow to upload bet slip screenshots (single or parlay), extract legs with OpenAI vision, let users correct parsing errors, then run the existing `computePropEvFields` pipeline per leg with a simple independent-leg parlay summary.

## Goal

- **Goal:** Ship `/betting/bet-slip-analyzer` with `POST /api/betting/bet-slip/parse` and `POST /api/betting/bet-slip/analyze`, reusing `OPENAI_API_KEY` / `OPENAI_MODEL` and EV logic from the rest of the app.
- **Inputs:** Multipart image; JSON body with confirmed legs + optional `resolved_player_id` for disambiguation.
- **Output/Done:** Working page + routes; MDX log with verification checklist.

## Implementation Notes (key logic explained)

- **Secrets:** OpenAI only on server; client calls `/api/betting/bet-slip/parse` with `FormData`.
- **Extraction:** Chat Completions with `image_url` data URL, `response_format: json_object`, Zod parse after stripping optional markdown fences.
- **EV:** Each leg builds `PropEvRowInput` with `market_type: 'over_under'`; `resolveEvTrack()` matches the rest of the app.
- **Player match:** Optional `resolved_player_id` skips fuzzy search when the user picks from ambiguous candidates.
- **Parlay:** `computeParlaySummary` multiplies leg `modelProbability` values when all legs analyzed; total implied P from American or decimal total odds; EV = `pCombined * D - 1` when `D` is known.

## Key Decisions (and alternatives considered)

- **Decision:** Raw `fetch` to OpenAI (same as AI projection summary) — no new SDK dependency.
- **Alternative:** Official `openai` package — rejected for MVP to stay consistent with existing routes.
- **Decision:** No persistence — state lives in React; paper bets require `gameId` per row and were not extended.

## Assumptions

- `analytics.players` has `first_name`, `last_name`, `full_name` for name resolution.
- Vision model (`gpt-4o-mini` by default) returns JSON matching the Zod schema when prompted.
- Over/under player props only; other market types are out of scope.

## Failure Modes

- Missing or invalid `OPENAI_API_KEY` → 503 `NO_OPENAI_KEY` on parse.
- Unreadable image → model may return empty legs or `needs_review`; user must edit manually.
- Ambiguous name match → user must pick a candidate or fix the name.
- Unsupported `prop_type` text → leg shows `propTypeError` without EV.

## How to Test (commands + steps)

1. `npm run build`
2. `npm run dev` — open `/betting/bet-slip-analyzer` (with `OPENAI_API_KEY` set in `.env`).
3. Upload a small JPEG/PNG slip; confirm legs populate in the review form.
4. Edit a leg intentionally wrong → Run analysis → confirm unmatched or wrong EV until corrected.
5. For a parlay with total odds on the slip, confirm parlay block shows combined P and estimated EV when all legs analyze.
6. Optional: call `POST /api/betting/bet-slip/analyze` with JSON body matching `betSlipAnalyzeBodySchema` (e.g. via curl or REST client) with a known `player_name` from your DB.
