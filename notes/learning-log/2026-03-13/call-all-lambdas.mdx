# Call All Lambdas Script

## Context

The repo has five Lambda handlers under `lambda/`: nightly-bdl-updater, odds-pre-game-snapshot, injuries-snapshot, player-props-snapshot, boxscore-scraper. Four are deployed via Terraform; boxscore-scraper is not. There was no single entrypoint to run or invoke all of them.

## Goal

Provide one way to "call all" Lambda functions: either run their handlers locally in sequence or invoke the deployed AWS functions.

## Files Changed

- **scripts/call-all-lambdas.ts** (new) – Script that runs all lambdas locally or invokes them on AWS.
- **package.json** – Added `lambdas` and `lambdas:aws` scripts.

## Implementation Notes

- **Local mode (default):** For each of the five lambda dirs, the script spawns `npx tsx index.ts` with `cwd` set to that lambda’s folder so a `.env` in `lambda/<name>/` is used. Runs in order: nightly-bdl-updater → odds-pre-game-snapshot → injuries-snapshot → player-props-snapshot → boxscore-scraper. Exits on first non-zero exit code.
- **AWS mode (`--aws`):** Invokes four deployed functions via `aws lambda invoke --function-name <name> --payload '{}' <outfile>`. Function names match Terraform example: nightly-bdl-updater, odds-pre-game-snapshot, injuries-snapshot, nba-player-props-ingestion-lambda. boxscore-scraper is not invoked (not in infra).
- Uses Node `child_process.spawn` with `shell: true` so `npx` and `aws` resolve correctly on Windows.

## Key Decisions

- **Local first, AWS optional:** Default is local so it works without AWS credentials; `--aws` is for deployed invokes.
- **No new dependencies:** Uses only Node built-ins (`child_process`, `path`, `url`).
- **Sequential execution:** Lambdas run one after another to avoid DB/API contention and to keep logs readable.

## Assumptions

- Each lambda has `npx tsx index.ts` working from its own directory (existing pattern).
- AWS CLI is installed and configured when using `--aws`.
- Terraform function names match `infra/terraform.tfvars.example` (or user passes correct names via env/CLI if we extend later).

## Failure Modes

- Missing `.env` or wrong `SUPABASE_DB_URL` / `BALLDONTLIE_API_KEY` in a lambda dir → that lambda fails locally.
- AWS credentials not configured or wrong region → AWS invokes fail.
- If a lambda dir is missing or has no `index.ts`, the spawn will fail (e.g. "command not found" or missing file).

## How to Test

**Local (all five):**

```bash
npm run lambdas
# or
npx tsx scripts/call-all-lambdas.ts
```

**AWS (four deployed):**

```bash
npm run lambdas:aws
# or
npx tsx scripts/call-all-lambdas.ts --aws
```

Ensure at least one lambda has a valid `.env` (e.g. `lambda/nightly-bdl-updater`) and run `npm run lambdas`; first lambda should run and then you can interrupt or fix others as needed.
