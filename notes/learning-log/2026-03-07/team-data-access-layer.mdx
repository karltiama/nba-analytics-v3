## Context
Final step in the team analytics data layer — shared types and query functions for the Team Page MVP. The DDL and compute script were created in prior steps.

## Goal
Create a typed data access layer that queries only `analytics.*` tables, ready to wire into a Team Page.

## Files Changed
- `lib/teams/types.ts` — **new** shared TypeScript interfaces
- `lib/teams/analytics-queries.ts` — **new** query functions

## Implementation Notes
- **`resolveAnalyticsTeamId`** — accepts either `team_id` or abbreviation (e.g. "LAL"), matches the pattern from `resolveAnalyticsPlayerId`.
- **`getTeamById`** — simple lookup from `analytics.teams`.
- **`getTeamSeasonAverages`** — returns latest season if no season specified. All numeric fields explicitly cast through `Number()` to avoid pg returning strings for `numeric` columns.
- **`getTeamRecentGames`** — joins opponent team info, supports optional season filter. Returns most-recent-first (for tables/lists).
- **`getTeamTrendData`** — lightweight projection for charts. Fetches most-recent-first from DB then `.reverse()`s to oldest-first for chart rendering (matching the player trend chart convention).

## Key Decisions
- **Separate `types.ts`** rather than inline types — the existing project has all team types inline in components. Centralizing them prevents drift as the team page grows.
- **`TeamTrendPoint` as a distinct type** rather than reusing `TeamGameStats` — keeps chart data minimal and decoupled from the full game row.
- **Optional `season` parameter** on all queries — supports both "current season" (default) and "historical" views without separate functions.
- **`opponent_abbr` and `opponent_name` joined in queries** rather than requiring a second lookup — fewer round trips, the Team Page needs these for display.

## Assumptions
- `analytics.team_game_stats` and `analytics.team_season_averages` are populated (run `compute-team-stats.ts` first).
- `pg` returns `numeric` columns as strings — all averages are cast through `Number()`.
- Team IDs in the URL may be either the raw ID or an abbreviation — `resolveAnalyticsTeamId` handles both.

## Failure Modes
- If the derived tables are empty, all functions return null / empty arrays gracefully.
- If a team has games but no opponent in `analytics.teams`, the JOIN will exclude those rows (unlikely but safe).

## How to Test
1. Ensure derived tables are populated: `npx tsx scripts/compute-team-stats.ts`
2. Test in a Next.js server component or API route:
   ```ts
   import { getTeamById, getTeamSeasonAverages, getTeamRecentGames, getTeamTrendData } from '@/lib/teams/analytics-queries';

   const team = await getTeamById('some-team-id');
   const avg = await getTeamSeasonAverages('some-team-id');
   const games = await getTeamRecentGames('some-team-id', 5);
   const trend = await getTeamTrendData('some-team-id', 20);
   ```
3. Verify trend data is oldest-first (for chart rendering).
4. Verify numeric fields are actual numbers, not strings.
