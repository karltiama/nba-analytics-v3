## Context
The betting player page has tabs: Trends, Matchup, Game Log. The Matchup tab previously showed a simple Next Game card, client-computed Recent Form (L5), Vs Opponent (filtered from games), and a "Coming Soon" placeholder. No opponent defensive context, no server-backed recent form or vs-opponent data.

## Goal
Implement the ideal Matchup tab: forward-looking analysis with Next Game Overview, Opponent Context (defensive rating, pace, points/rebounds allowed), Player Recent Form (L5 + minutes vs season), Vs Opponent History (server data), and a future Odds placeholder—using only analytics schema and existing patterns.

## Implementation Notes (key logic explained)
- **Opponent context:** Defensive rating, pace, points allowed come from `getTeamSeasonAverages(opponentTeamId)`. Rebounds allowed = AVG(opponent_offensive_rebounds + opponent_defensive_rebounds) in `analytics.team_game_stats` where `team_id = opponentTeamId`. Assists allowed not in schema; UI shows "—" with tooltip.
- **Recent form:** Single query with CTE: last N rows from player_game_logs (Final games, order by game_date DESC), then AVG(points), AVG(rebounds), AVG(assists), AVG(pra), AVG(minutes). Minutes parsed with regex for numeric-only values.
- **Vs opponent:** Query player_game_logs where `player_id` and `opponent_team_id` (and optional season); order by game_date ASC for chart-ready order; return aggregates + full `games` list.
- **Days of rest:** Computed client-side: difference in days between `games[0].start_time` (last game) and `nextGame.start_time`.

## Key Decisions (and alternatives considered)
- **Server-side recent form and vs-opponent:** Plan asked for query helpers; page fetches them and passes to tab so Matchup tab is server-data driven. Client could still derive from `games` as fallback; we use server data only.
- **Season avg minutes:** Not in `analytics.player_season_averages`; computed from `games` array in the tab (average of `games[].minutes`). Documented for future schema addition.
- **Vs opponent chart:** Plan said "optional mini trend … if reuse is awkward, skip the chart." We did not add the chart; stats grid only. Chart can be added later using `vsOpponentHistory.games` and `PlayerTrendChart`.

## Assumptions
- Next game and current team from analytics only (`getNextGameForPlayer`). Season for opponent/vs history = `nextGame.season` when next game exists.
- `analytics.player_game_logs.minutes` is text; only numeric values (e.g. `^[0-9]+\.?[0-9]*$`) are used for AVG(minutes) in recent form.
- When there is no next game, Opponent Context and Vs Opponent History show "No next game to compare" or equivalent; Recent Form and placeholder still render.

## Failure Modes
- If `getTeamSeasonAverages` returns null for the opponent, `getOpponentContextForGame` returns null; UI shows "Opponent team stats not available."
- If player has no games, `recentForm.games_played === 0`; UI shows "No recent games."
- Vs opponent 0 games: "No games vs this opponent this season."

## How to Test (commands + steps)
1. Run app: `npm run dev`. Open a player with upcoming game and past games (e.g. `/betting/players/<id>?season=2025`).
2. Open Matchup tab. Confirm: Next Game Overview (vs/@, date, home/away, days rest, matchup label), Opponent Context (def rating, pace, pts allowed, reb allowed), Recent Form (L5 PTS/REB/AST/PRA/MIN, season minutes if available), Vs Opponent (sample size + averages), Odds Analysis placeholder.
3. Player with no next game: Next Game shows "No upcoming game scheduled"; Opponent Context and Vs Opponent show no-next-game message; Recent Form and placeholder still visible.
4. Player with no games: Recent Form shows "No recent games."
