## Context
The app had basic team derived tables (`analytics.team_game_stats`, `analytics.team_season_averages`) and a Team Page using analytics-only queries. Advanced metrics (ORtg, DRtg, Pace, eFG%, TOV%, ORB%) and home/away splits were not stored or displayed. Schedule and some components still used legacy `bbref_games` / `teams` (public).

## Goal
Add advanced team metrics using only the analytics schema: extend derived tables, compute metrics at team-game level, aggregate into season averages, expose via data layer, and show on the Team Page. Refactor schedule to use `analytics.games`.

## Files Changed
- `db/schemas/analytics_team_stats_advanced.sql` — **new** migration (ORB/DRB, opponent stats, advanced columns; home/away + advanced on TSA).
- `scripts/compute-team-stats.ts` — CTE aggregation with opponent join, formulas for estimated_possessions, ORtg, DRtg, pace, eFG%, TOV%, ORB%; season agg with home_wins/home_losses/away_wins/away_losses and AVG(advanced).
- `lib/teams/types.ts` — extended `TeamSeasonAverages` (home_wins, away_wins, avg_offensive_rating, etc.); new `TeamAdvancedMetrics`.
- `lib/teams/analytics-queries.ts` — extended `getTeamSeasonAverages`; added `getTeamAdvancedMetrics`.
- `lib/analytics/games-queries.ts` — added `getScheduleForTeam` and `ScheduleGameRow` for analytics.games schedule.
- `app/teams/[teamId]/page.tsx` — no change (already uses extended seasonAverages).
- `app/teams/[teamId]/components/TeamPageContent.tsx` — home/away splits in header; RPG/APG pills; Advanced metrics section (ORtg, DRtg, Pace, eFG%, TOV%, ORB%).
- `app/teams/[teamId]/components/TeamSchedule.tsx` — refactored to `resolveAnalyticsTeamId` + `getScheduleForTeam` (analytics only).
- `app/teams/[teamId]/schedule/page.tsx` — refactored to analytics: `resolveAnalyticsTeamId`, `getTeamById`, `getScheduleForTeam`; title "Schedule" (no BBRef).

## Implementation Notes (key logic explained)
- **Team-game aggregation**: A CTE `team_per_game` aggregates `player_game_logs` by `(game_id, team_id)` including `offensive_rebounds` and `defensive_rebounds`. A second CTE `with_opponent` joins `team_per_game` to itself on `game_id` and `opponent_team_id` so each row has both team and opponent stats. Advanced metrics are computed in the outer SELECT (estimated_possessions, then ORtg/DRtg/pace/efg_pct/tov_pct/orb_pct).
- **Formulas**: estimated_possessions = 0.5 * (team_poss_estimate + opponent_poss_estimate) with poss_estimate = FGA + 0.44*FTA - ORB + TOV. ORtg = (team_points / poss) * 100; DRtg = (points_allowed / poss) * 100; pace = estimated_possessions; efg_pct = (FGM + 0.5*3PM)/FGA; tov_pct = TOV/(FGA+0.44*FTA+TOV); orb_pct = ORB/(ORB+opp_DRB).
- **Schedule**: `getScheduleForTeam(teamId, season?)` queries `analytics.games` joined to `analytics.teams` for home/away names and abbrs; returns rows with is_home, opponent_id, scores, result. Team schedule page and TeamSchedule component resolve `teamId` via `resolveAnalyticsTeamId` then call this.

## Key Decisions (and alternatives considered)
- **Advanced metrics on season averages only in UI**: Per-game values are stored in `team_game_stats` but the Team Page only shows season averages from `team_season_averages`; optional per-game columns can be added to the game log later.
- **Single getTeamSeasonAverages for header + advanced**: No separate fetch for `getTeamAdvancedMetrics` on the page; extended `getTeamSeasonAverages` provides all fields. `getTeamAdvancedMetrics` exists for callers that only need the six metrics.
- **Schedule refactor**: Replaced `bbref_games` + public `teams` with `analytics.games` + `analytics.teams` and new `getScheduleForTeam`; links use `opponent_id` (analytics team_id) so `/teams/[id]` works with analytics resolution.

## Assumptions
- Migration `analytics_team_stats_advanced.sql` is run before re-running `compute-team-stats.ts`.
- `analytics.player_game_logs` has `offensive_rebounds` and `defensive_rebounds` populated.
- Schedule page and TeamSchedule are only used with a `teamId` that can be resolved via `resolveAnalyticsTeamId` (analytics team_id or abbreviation).

## Failure Modes
- If the migration has not been applied, the compute script will fail on insert (unknown columns).
- If a game has only one team in `player_game_logs`, the opponent join will drop that row (no row for that team in that game); ensure both teams have logs.
- Null `start_time` in analytics.games: schedule UI shows "TBD" for date/time and treats game as upcoming.

## How to Test (commands + steps)
1. Run migration: `psql $SUPABASE_DB_URL -f db/schemas/analytics_team_stats_advanced.sql`
2. Run compute script: `npx tsx scripts/compute-team-stats.ts`
3. In DB: `SELECT team_id, season, avg_offensive_rating, avg_defensive_rating, avg_pace, home_wins, away_wins FROM analytics.team_season_averages LIMIT 5;`
4. Open Team Page for a team (e.g. `/teams/BOS` or `/teams/1610612738`): confirm record, PPG, Opp PPG, RPG, APG, Home/Away splits, and Advanced metrics section.
5. Open `/teams/BOS/schedule`: confirm schedule loads from analytics (no BBRef), past/upcoming split, and links to opponent team pages.
