## Context
The team page (`/teams/[teamId]`) was a client component that fetched from 4 API routes, all backed by legacy `public.*` and `bbref_*` tables. Rewritten to use only `analytics.*` schema, matching the server-component pattern from the betting player page.

## Goal
Wire the Team Page MVP to the new `analytics.team_game_stats` and `analytics.team_season_averages` tables. Remove all legacy data dependencies.

## Files Changed
- `app/teams/[teamId]/page.tsx` — **rewritten** as server component; loads data via analytics queries, passes as props
- `app/teams/[teamId]/components/TeamPageContent.tsx` — **rewritten** as `TeamPageClient`; receives data as props, no more `fetch` calls
- `app/teams/[teamId]/components/RecentGamesTimeline.tsx` — **updated** interface to use `TeamGameStats` from analytics types
- `app/teams/[teamId]/components/TeamRoster.tsx` — **rewritten** to derive roster from `analytics.player_game_logs` instead of legacy `player_team_rosters`
- `lib/teams/types.ts` — **updated** added `is_home` to `TeamTrendPoint`
- `lib/teams/analytics-queries.ts` — **updated** `getTeamTrendData` now includes `is_home`

## Implementation Notes
- **Server component pattern**: `page.tsx` calls `resolveAnalyticsTeamId` → parallel `Promise.all` for team info, season averages, recent games, trend data → passes to client component for interactivity.
- **Roster derived from game logs**: Since analytics schema has no roster table, we derive it from `analytics.player_game_logs` grouped by `(player_id, team_id, season)` with a game count. Sorted by games played DESC.
- **Location filter**: `TeamTrendPoint` now includes `is_home` so the chart's home/away filter works directly without cross-referencing `recentGames`.
- **Trend data is oldest-first** (`.reverse()` in the query function), slicing uses `slice(-N)` to take the last N entries.

## Key Decisions
- **Dropped AI insights and upcoming games** for now — no analytics data source for these. Placeholder "Coming Soon" section kept.
- **Roster shows games played** instead of jersey number (no jersey data in analytics schema). Displayed as `{count}g`.
- **No API routes needed** — server component queries DB directly, matching the betting player page pattern. Removes the client-side fetch waterfall.

## Assumptions
- The team's URL parameter can be either a raw `team_id` or an abbreviation — `resolveAnalyticsTeamId` handles both.
- Season is hardcoded to `'2025'` for roster; will need to be dynamic when multi-season support is added.
- `TeamTrendChart.tsx` was not modified — it's a pure chart component that's data-agnostic.

## Failure Modes
- If `compute-team-stats.ts` hasn't been run, the page will show "Team not found" (no data in derived tables, though `analytics.teams` would still resolve).
- If a player has logs for multiple teams in a season, they'll appear on both rosters (correct — reflects mid-season trades).

## How to Test
1. Navigate to `/teams/{any-team-id}` — should load server-side with no client fetches
2. Verify team header shows record, win%, PPG, Opp PPG, streak
3. Verify trend chart renders with metric/timeframe/location filters working
4. Verify "Last 5 Games" table shows correct scores, opponents, margins
5. Verify roster sidebar shows players with games played count
6. Check browser Network tab — no `/api/teams/*` calls should appear
