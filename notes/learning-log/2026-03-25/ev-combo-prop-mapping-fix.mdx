## Context
Explorer rows showed implausibly high EV (often >30%) on combo props, especially `under`, with projections far below line.

## Goal
Fix combo prop stat mapping so projection inputs for combo markets use the correct aggregated stats.

## Files Changed
- `lib/betting/player-prop-inputs.ts`
- `lib/betting/player-prop-model.ts`

## Implementation Notes (key logic explained)
- Extended model stat inputs with combo aggregates:
  - `pa` = points + assists
  - `pr` = points + rebounds
  - `ra` = rebounds + assists
- Added season/last10/last5/std10 computations for `pa`, `pr`, `ra` from game logs.
- Added direct prop mappings:
  - `points_assists` -> `pa`
  - `points_rebounds` -> `pr`
  - `rebounds_assists` -> `ra`
- Added fallback inference for combo aliases in `inferStatFromPropType`.
- Added combo fallback sigma defaults in `player-prop-model.ts` for these market types.

## Key Decisions (and alternatives considered)
- Decision: correct upstream stat mapping rather than just capping EV.
- Alternative: hard EV cap in API.
  - Rejected because root cause is wrong projection basis for combo props.

## Assumptions
- Combo prop lines are over/under on sum stats and should be modeled with sum distributions.

## Failure Modes
- Correlated stat variance remains approximated from empirical game-level sums (acceptable for this model tier).
- Rare prop-type naming variants may still need additional alias entries.

## How to Test (commands + steps)
1. `npm run build`
2. Query `/api/betting/props-explorer?date=<today>&sort=ev&dir=desc&limit=200`
3. Confirm combo prop projections (`points_assists`, `points_rebounds`, `rebounds_assists`) are near plausible ranges.
4. Confirm top `EV > 30%` count decreases materially.
