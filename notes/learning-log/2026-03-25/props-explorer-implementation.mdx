## Context

The betting stack now has reliable multi-book props in `analytics.player_props_current` and a dual-track EV pipeline (Track B default, Track A reference). A slate-wide explorer was missing for validation and quick navigation.

## Goal

Ship `/betting/props-explorer` with server-side filters and pagination, EV columns aligned with the player props API, and shareable URL query params.

- [`lib/betting/player-prop-ev-row.ts`](lib/betting/player-prop-ev-row.ts) — shared `computePropEvFields` for over/under rows.
- [`app/api/betting/players/[playerId]/props/route.ts`](app/api/betting/players/[playerId]/props/route.ts) — uses shared helper (no formula drift).
- [`app/api/betting/props-explorer/route.ts`](app/api/betting/props-explorer/route.ts) — new list endpoint.

- [`components/betting/Header.tsx`](components/betting/Header.tsx) — Dashboard + Props Explorer nav links.

## Implementation Notes (key logic explained)

- Explorer SQL joins `player_props_current` to `analytics.games` and filters by ET calendar day (same window as `getGamesForDate`) unless `game_id` is set.
- EV sorts (`ev`, `ev_track_a`, `ev_track_b`) load up to `EV_FETCH_CAP` freshest rows, compute EV in Node, optionally filter `min_ev`, sort, then slice for `offset`/`limit` — global EV order is approximate on huge slates by design (MVP).
- Non-EV sorts use SQL `ORDER BY` + true `LIMIT`/`OFFSET`.
- Model inputs are loaded per unique `player_id` in the fetched batch (chunked parallel `getPlayerPropModelInputs`).

## Key Decisions (and alternatives considered)

- **Dedicated API** vs client-only: chosen for scale and consistent JSON shape.
- **EV sort cap** vs full-table sort: cap keeps latency bounded; documented in UI when active.

## Assumptions

- Explorer users accept approximate global ranking when sorting by EV on very large dates.
- `game_id` in the UI matches `analytics.player_props_current.game_id`.

## Failure Modes

- Heavy dates may slow the EV-sort path despite the cap.
- Rows without model inputs show null EV/model columns (same as player page).
- `analytics.games.game_id` is `text` while `player_props_current.game_id` may be `integer` — the explorer join uses `g.game_id = p.game_id::text` to avoid `text = integer` errors.

## How to Test (commands + steps)

1. `npm run build`
2. Open `/betting/props-explorer` — table loads for today (ET).
3. Change date, prop type prefix, side, sportsbook — URL updates and data refetches.
4. Sort by EV — confirm ordering changes; note cap warning banner.
5. `GET /api/betting/props-explorer?date=YYYY-MM-DD&limit=5` — valid JSON with `rows` and `meta`.
