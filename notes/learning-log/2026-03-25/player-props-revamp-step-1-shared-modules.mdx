## Context
The player props ingestion Lambda was timing out before writing UI-facing tables. The first step is to split core logic into reusable modules and prepare separate controller/worker handlers.

## Goal
Extract shared ingestion primitives (env, DB, game discovery, API fetch, normalization, bulk writes, metrics) so fan-out processing can be implemented cleanly.

## Implementation Notes (key logic explained)
- Added typed payload contracts for game targets and SQS worker messages.
- Added controller entrypoint to enqueue one message per game into SQS and initialize pull/game run records.
- Added worker entrypoint to process per-game payloads and use chunked bulk DB operations.
- Replaced monolithic `index.ts` body with a simple export alias to controller handler.
- Implemented chunked insert/upsert functions for `raw.player_prop_snapshots_v2` and `analytics.player_props_current`.
- Added helper to refresh legacy `analytics.player_prop_current` lines for preferred vendor compatibility.

## Key Decisions (and alternatives considered)
- Chose module extraction before infra rewiring so logic can be reused by both handlers.
- Chose SQL chunked VALUES statements over row-by-row queries to reduce DB round-trips.
- Kept legacy `analytics.player_prop_current` refresh in worker to avoid breaking existing game endpoint consumers.

## Assumptions
- `raw.player_prop_game_runs` table will be added in a later diff for per-game status updates.
- SQS queue URL is injected via `PLAYER_PROPS_QUEUE_URL` in Lambda environment.

## Failure Modes
- Missing queue URL in controller environment will throw on startup.
- Missing new DB table (`raw.player_prop_game_runs`) will fail worker writes until schema migration is applied.
- Large chunk sizes could still pressure DB if runtime settings are too low.
