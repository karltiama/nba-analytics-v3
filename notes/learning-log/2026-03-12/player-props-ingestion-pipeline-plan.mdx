# Player Props Ingestion Pipeline (Plan Implementation)

## Context

The NBA analytics app already had a player-props Lambda (`lambda/player-props-snapshot`) that fetches BallDontLie `/v2/odds/player_props` per game and writes to `raw.player_prop_snapshots` (legacy) and `analytics.player_prop_current`. That Lambda was not in Terraform, and the analytics shape was one row per market (over_odds/under_odds columns) rather than one row per side with American/decimal/implied probability for a Prop Explorer. The plan was to add Terraform (Lambda + EventBridge Scheduler), new raw and analytics tables (v2 shape), and extend the Lambda to write to both legacy and v2 tables.

## Goal

- Terraform: Lambda `nba-player-props-ingestion-lambda`, IAM roles (Lambda execution + Scheduler invoke), EventBridge Scheduler `nba-player-props-schedule` (every 30 min, flexible time window OFF).
- SQL: `raw.player_prop_snapshots_v2` (append-only, flattened per sportsbook/prop/side with odds_american, odds_decimal, implied_probability) and `analytics.player_props_current` (latest rows for Prop Explorer).
- Lambda: Helpers (convertAmericanToDecimal, calculateImpliedProbability, normalizePlayerPropResponse, insertRawSnapshots, upsertAnalyticsCurrent, fetchTodaysGames); handler writes to v2 raw + analytics after existing legacy path.

## Files Changed

- `db/schemas/raw_player_prop_snapshots_v2.sql` — new raw table and indexes.
- `db/schemas/analytics_player_props_current.sql` — new analytics table and unique + indexes.
- `lambda/player-props-snapshot/index.ts` — odds helpers, NormalizedPropRow, normalizePlayerPropResponse, insertRawSnapshots, upsertAnalyticsCurrent, fetchTodaysGames; processDate collects BDL rows and runs v2 insert + upsert with structured logging.
- `infra/variables.tf` — player_props_* variables (function name, timeout, memory, env, enable_schedule, schedule_expression).
- `infra/iam.tf` — lambda_player_props_execution role, scheduler_player_props_invoke role and inline policy (lambda:InvokeFunction).
- `infra/lambda.tf` — archive_file.player_props, aws_lambda_function.player_props_ingestion, aws_scheduler_schedule.player_props, aws_lambda_permission.allow_scheduler_player_props.
- `infra/outputs.tf` — player_props_lambda_function_name, player_props_lambda_function_arn, player_props_schedule_name.
- `infra/terraform.tfvars.example` — player_props_* example block.
- `lambda/player-props-snapshot/README.md` — pipeline diagram updated (v2 tables, Terraform section).

## Implementation Notes

- **Odds conversion:** Negative American: decimal = 1 + 100/|odds|, implied = |odds|/(|odds|+100). Positive: decimal = 1 + odds/100, implied = 100/(odds+100).
- **Normalize:** Each BDL over_under row becomes two NormalizedPropRow (side over/under); milestone becomes one row (side milestone). raw_json stores the original BDL row for debugging.
- **Raw v2:** Single table, append-only; no unique constraint. Bulk insert per row with try/catch to log DB failures without failing the whole run.
- **Analytics upsert:** ON CONFLICT (game_id, player_id, sportsbook, prop_type, side, line_value) DO UPDATE to keep latest snapshot_at and odds columns.
- **EventBridge Scheduler:** Uses `aws_scheduler_schedule` with target.role_arn = IAM role that has lambda:InvokeFunction on the Lambda; Lambda permission principal = scheduler.events.amazonaws.com.

## Key Decisions

- **Option A (new tables):** Added `raw.player_prop_snapshots_v2` and `analytics.player_props_current` alongside existing tables so the game page and existing API keep working; Lambda writes to both legacy and v2.
- **SUPABASE_DB_URL:** Kept same as other Lambdas (direct Postgres) in Terraform/tfvars.example; spec’s SUPABASE_URL/SUPABASE_SERVICE_ROLE_KEY documented as alternative for Supabase client use elsewhere.
- **EventBridge Scheduler vs CloudWatch Events:** Plan asked for EventBridge Scheduler (rate(30 minutes), flexible time window OFF); implemented with aws_scheduler_schedule. Optional comment in Terraform for adding a pregame-only schedule later.

## Assumptions

- BDL response shape unchanged (over_under with over_odds/under_odds, milestone with odds); player_name/team_id not in BDL response so stored as null in v2.
- analytics.games exists and game_id is numeric-parsable (used for today’s games query).
- Terraform zip packages the whole lambda/player-props-snapshot dir (including dist/); user runs `npm run build` before apply.

## Failure Modes

- Missing SUPABASE_DB_URL or BALLDONTLIE_API_KEY: Lambda throws at cold start.
- BDL 429: Existing retry (60s wait) in fetchPlayerPropsForGame.
- DB insert/upsert failure: Logged per row; raw v2 insert and analytics upsert continue for remaining rows.
- Scheduler not invoking: Check Lambda permission for scheduler.events.amazonaws.com and Scheduler role has InvokeFunction on the Lambda.

## How to Test

1. **SQL:** Run `db/schemas/raw_player_prop_snapshots_v2.sql` and `db/schemas/analytics_player_props_current.sql` against the DB (after raw_schema and analytics_schema).
2. **Lambda locally:** `cd lambda/player-props-snapshot && npx tsx index.ts` (with .env SUPABASE_DB_URL, BALLDONTLIE_API_KEY). Check logs for "V2 — raw snapshots inserted: N, analytics.player_props_current upserted: M".
3. **Terraform:** `cd infra && terraform init && terraform validate`. Optionally `terraform plan -var-file=terraform.tfvars` (with player_props_lambda_env set); ensure player-props zip is built first.
4. **Query analytics:** `SELECT * FROM analytics.player_props_current WHERE player_id = <id> ORDER BY prop_type, line_value, sportsbook;`
