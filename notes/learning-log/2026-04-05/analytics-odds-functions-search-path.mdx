## Context

Supabase (or `pglint`) reported `analytics.american_to_decimal` has a mutable `search_path` — a known hardening issue for SECURITY DEFINER–style risks when unqualified names resolve in caller-dependent order.

## Goal

- **Goal:** Pin `search_path` on SQL helper functions in `analytics`.
- **Inputs:** `db/schemas/analytics_player_prop_lines_schema.sql`.
- **Output/Done:** `SET search_path = ''` on both helpers; one-off migration for existing DBs.

## Files Changed

- `db/schemas/analytics_player_prop_lines_schema.sql` — `set search_path = ''` on `american_to_decimal` and `american_to_implied_prob`.
- `db/schemas/MIGRATION_analytics_odds_functions_search_path.sql` — apply in Supabase if objects already deployed.

## Implementation Notes (key logic explained)

- Bodies use only literals and built-in ops; empty `search_path` forces resolution via `pg_catalog` only, which is appropriate here.
- **Alternative:** `SET search_path = 'analytics', 'pg_catalog'` if you later add unqualified refs to `analytics` objects inside the function.

## How to Test

1. In Supabase SQL Editor, run `MIGRATION_analytics_odds_functions_search_path.sql`.
2. Re-run linter / `SELECT analytics.american_to_decimal(100);` → expect `2`.

## Verification Checklist

1. Migration applies without error.
2. Linter no longer flags these functions.
3. Existing transforms using `analytics.american_to_decimal` still behave the same.
