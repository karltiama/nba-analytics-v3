## Context

Production (e.g. Vercel) could not reach Postgres while local dev worked, even when `SUPABASE_DB_URL` appeared to match. Lambdas in this repo already used `ssl` and trimmed URLs; `lib/db.ts` did not.

## Goal

Align Next.js server Postgres (`lib/db.ts`) with proven Supabase connection handling: trim the connection string and enable TLS for Supabase hosts.

## Files Changed

- `lib/db.ts`

## Implementation Notes (key logic explained)

- **`trim()`** on `SUPABASE_DB_URL`: hosting UIs sometimes store a trailing newline or space. That can corrupt the hostname inside `pg`’s parser and surface as **`getaddrinfo ENOTFOUND`** (DNS lookup for a bad host string).
- **`ssl: { rejectUnauthorized: false }`** when the URL looks like Supabase: matches `lambda/odds-pre-game-snapshot/index.ts` and other workers; Supabase uses certificates that `pg` often needs this flag for (same pattern as `scripts/fetch-sample-player-props.ts`).

## Key Decisions (and alternatives considered)

- **Conditional SSL** only for `supabase.co` / `pooler.supabase.com` so non-Supabase Postgres URLs are unchanged.
- **Alternative:** append `?sslmode=require` in the dashboard connection string only; still recommend `trim()` in code for newline issues.

## Assumptions

- Production DB is Supabase-hosted Postgres accessed via URI in `SUPABASE_DB_URL`.

## Update: `ENOTFOUND` for `db.<project>.supabase.co` on Vercel (IPv6 vs IPv4)

Supabase **direct** DB hostnames (`db.*.supabase.co:5432`) are **IPv6-first**; public DNS often returns **only AAAA** (no IPv4 `A` record). Many local networks support IPv6, so dev works. **Vercel serverless** commonly resolves/connects over **IPv4-only**, so the same hostname can fail DNS or connectivity with **`getaddrinfo ENOTFOUND`** or similar.

**Fix (recommended):** In Supabase Dashboard → **Connect** → use **Session pooler** (or the shared pooler URI that uses `aws-0-<region>.pooler.supabase.com`). That hostname has **IPv4** `A` records and is intended for environments without IPv6. Paste that full URI into **`SUPABASE_DB_URL`** on Vercel and redeploy.

**Alternatives:** Supabase **IPv4 add-on** for direct connections ([docs](https://supabase.com/docs/guides/platform/ipv4-address)); or **Transaction pooler** (`:6543`) for serverless—see Supabase docs on **prepared statements** with `node-pg` if you use transaction mode.

## Failure Modes

- If the error is still **`ENOTFOUND`** after switching to the **pooler** host, verify the **exact** production string (no typo; correct Vercel project and **Production** environment); `trim()` fixes whitespace, not wrong hostnames.
- If you use a **custom** Postgres host that needs SSL but does not match the substring check, set SSL via env or extend the condition.
