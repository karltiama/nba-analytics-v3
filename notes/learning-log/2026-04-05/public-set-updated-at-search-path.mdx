## Context

Supabase linter flagged `public.set_updated_at` for mutable `search_path` (same class of issue as unqualified resolution in triggers).

## Goal

- **Goal:** Pin `search_path` on the shared `updated_at` trigger function.
- **Inputs:** `db/schemas/MIGRATION_user_system_mvp.sql`.
- **Output/Done:** `set search_path = ''` on `public.set_updated_at`; one-off migration for live DBs.

## Files Changed

- `db/schemas/MIGRATION_user_system_mvp.sql` — function definition updated for new installs.
- `db/schemas/MIGRATION_set_updated_at_search_path.sql` — apply on existing projects.

## Implementation Notes (key logic explained)

- Body only uses `now()` and `NEW`; empty `search_path` keeps resolution to built-ins / trigger protocol.

## How to Test

1. Run `MIGRATION_set_updated_at_search_path.sql` in Supabase SQL Editor.
2. `UPDATE public.profiles SET display_name = display_name WHERE false;` (no-op) or a real update — `updated_at` should still advance when rows change.
3. Re-run linter — warning should clear.

## Verification Checklist

1. Migration applies without error.
2. Triggers on `profiles`, `user_settings`, `user_saved_props` still fire on `UPDATE`.
