## Context

The app already had `profiles`, `user_settings`, and `/api/user/profile` + `/api/user/settings`, but no first-run onboarding. We added a two-step modal (sportsbook, odds format, paper display; then optional goal and experience) and persistence via a dedicated API plus schema columns.

## Goal

- **Goal:** Show a blocking onboarding modal on `/betting/*` for authenticated users who have not completed onboarding; save answers to the database and mark completion.
- **Inputs:** Authenticated session (Supabase cookie or bearer); form fields from the modal.
- **Output/Done:** `onboarding_completed_at` set on `profiles`; extended `user_settings` row; modal dismisses; `GET /api/user/profile` exposes `onboardingCompletedAt`.

## Files Changed

- `db/schemas/MIGRATION_onboarding_user_settings.sql` — new columns + reference to manual backfill script.
- `db/scripts/backfill-onboarding-completed.sql` — optional one-time prod backfill for existing accounts.
- `app/api/user/profile/route.ts` — `onboarding_completed_at` on reads and `PUT` `RETURNING`.
- `app/api/user/onboarding/route.ts` — `POST` to upsert settings and set `onboarding_completed_at`.
- `app/api/user/settings/route.ts` — `GET`/`PUT` `RETURNING` + `mapSettings` include new fields.
- `lib/onboarding/sportsbooks.ts` — searchable sportsbook list.
- `components/betting/OnboardingModal.tsx` — Radix `Dialog` UI (two steps).
- `components/betting/OnboardingGate.tsx` — fetches profile; shows modal when `onboardingCompletedAt` is null.
- `components/betting/index.ts` — exports.
- `app/betting/layout.tsx` — wraps content with `OnboardingGate`.

## Implementation Notes (key logic explained)

- **Completion flag:** `profiles.onboarding_completed_at` is the source of truth. `OnboardingGate` calls `GET /api/user/profile` with `credentials: 'include'`. If `401`, no modal. If `onboardingCompletedAt` is `null`, show modal.
- **Save path:** `POST /api/user/onboarding` ensures a `profiles` row, upserts `user_settings` with explicit `EXCLUDED` columns (not the old `COALESCE`-merge pattern), then sets `onboarding_completed_at = now()`.
- **Modal UX:** Overlay and Escape do not dismiss until Finish succeeds (required onboarding). Step 2 optional fields may be null.
- **Backfill:** The migration does **not** auto-update existing rows (safe re-runs). For production, run `db/scripts/backfill-onboarding-completed.sql` **once** if you want legacy accounts to skip the modal without completing the flow.

## Key Decisions (and alternatives considered)

- **Dedicated `POST /api/user/onboarding`** instead of two client calls (`PUT` settings + `PUT` profile): one atomic user action, simpler client, avoids partial state if one request fails.
- **No new npm dependencies:** `radix-ui` was already a dependency; `Dialog` used for focus and portal behavior.
- **Alternatives:** Local-only onboarding (localStorage) would not sync across devices; skipped.

## Assumptions

- `SUPABASE_DB_URL` and auth env vars are configured; migration is applied before relying on new columns.
- Users who should see the modal are logged in with a valid session cookie to `/api/user/profile`.

## Failure Modes

- **Migration not applied:** Profile/settings queries error; modal stays hidden (`OnboardingGate` treats non-OK profile fetch as no modal).
- **Not logged in:** `401` on profile; no modal (anonymous `/betting` visitors are unchanged).
- **Re-running backfill script:** Running `backfill-onboarding-completed.sql` more than once is harmless (idempotent `WHERE onboarding_completed_at IS NULL` only updates remaining nulls).

## How to Test (commands + steps)

1. Apply `db/schemas/MIGRATION_onboarding_user_settings.sql` in the Supabase SQL editor (or your migration runner).
2. (Optional) For existing accounts, run `db/scripts/backfill-onboarding-completed.sql` once if you do not want them to see the modal.
3. `npm run build` — should pass.
4. `npm run dev`, sign in, open `/betting` with a user whose `onboarding_completed_at` is `null` — modal should appear.
5. Complete the flow — confirm `profiles.onboarding_completed_at` and `user_settings` columns in the DB.
6. Reload `/betting` — modal should not reappear.
