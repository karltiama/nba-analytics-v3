## Context

The matchup **AI Projection Summary** card already showed deterministic chips (`getGameSummaryBullets`) and optional injury ÔÇťgame contextÔÇŁ prose. The product asked for a short **LLM-written** 2ÔÇô4 sentence summary grounded in that same on-page data.

## Goal

- **Goal:** Call OpenAI from the server to produce a 2ÔÇô4 sentence matchup summary from existing signals (bullets, optional injury narrative, optional odds hint).
- **Inputs:** `OPENAI_API_KEY` (optional); POST body with team names, bullets, optional injury fields, optional odds string.
- **Output/Done:** New API route returns `{ summary }`; the card shows loading, then text, or a clear fallback when the key is missing or the request fails.

## Files Changed

- `app/api/betting/games/[gameId]/ai-projection-summary/route.ts` ÔÇö `POST`, Zod validation, `fetch` to OpenAI Chat Completions (no new npm dependency).
- `components/betting/MatchupPageLayout.tsx` ÔÇö `formatOddsHintForAi`, client fetch on load, UI for loading / success / unavailable / error, footer copy when summary succeeds.

## Implementation Notes (key logic explained)

- **Server-only secret:** The browser never sees `OPENAI_API_KEY`; the route returns `503` with `code: "NO_OPENAI_KEY"` when unset so the UI can show setup instructions instead of a hard error.
- **No new package:** OpenAI is called via `fetch` to `https://api.openai.com/v1/chat/completions` (alternative: official `openai` SDK for retries and typings; adds a dependency).
- **Prompt:** System message instructs 2ÔÇô4 sentences, facts-only from the payload, no picks. User message includes matchup label, bullets, optional odds line, optional injury intro/paragraphs, plus `game id` for traceability.
- **Model:** Defaults to `gpt-4o-mini`; override with `OPENAI_MODEL` for cost or quality tuning.
- **Client:** One `useEffect` per loaded `data` builds the POST body from the same `getGameSummaryBullets` / `injuryNarrative` already used for display; `AbortController` cancels on unmount or Strict Mode double-invoke.

## Key Decisions (and alternatives considered)

- **POST with client-built context vs. server re-fetching `/details`:** Re-fetching would duplicate DB work and coupling; sending the structured context the UI already computed keeps one source of truth and fewer round trips.
- **`fetch` vs. `openai` package:** Zero dependency and small surface; tradeoff is manual error parsing and no built-in retries.
- **Graceful 503 when key missing:** Avoids treating ÔÇťnot configuredÔÇŁ as a user-facing failure; devs see how to enable.

## Assumptions

- Game details load as a single `data` object; refetching the page is acceptable if you later add live updates and want a fresh summary.

## Failure Modes

- Missing/invalid API key Ôćĺ 503; UI explains `OPENAI_API_KEY`.
- OpenAI outage or bad response Ôćĺ 502 / empty content; UI shows a short retry message.
- User disables network mid-request Ôćĺ aborted fetch; effect cleanup avoids setState after unmount (guard + `cancelled` flag).

### Update (server-side cache)

- **Route:** `unstable_cache` wraps the OpenAI call; cache key = `gameId` + SHA-256 of validated POST body + model; TTL from `AI_PROJECTION_SUMMARY_CACHE_SECONDS` (default **900**, min 60, max 86400).
- **Effect:** Identical refreshes reuse the cached summary and do not re-call OpenAI until the entry expires. Changing injuries/odds context changes the body hash Ôćĺ new call.
- **Limits:** Does not stop someone from sending *different* bodies to burn quota; combine with OpenAI spending limits and optional per-IP rate limiting for public deployments.
