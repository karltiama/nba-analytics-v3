## Context

Terminal showed `GET /signup?next=... 404` and repeated `/api/user/profile` 401 while onboarding linked to non-existent auth pages.

## Goal

- **Goal:** Ship `/login` and `/signup` with Supabase email/password and `next` redirect; refresh session cookies so `/api/user/*` can authenticate.
- **Inputs:** `next` query (validated internal path).
- **Output/Done:** Real pages, middleware session refresh, optional email-confirm callback.

## Files Changed

- `package.json` — `@supabase/supabase-js`, `@supabase/ssr`.
- `middleware.ts` — Supabase `updateSession` on matched routes.
- `lib/supabase/client.ts` — browser client.
- `lib/supabase/server.ts` — server client for route handler.
- `lib/supabase/middleware.ts` — `getUser()` cookie refresh.
- `lib/auth/safe-next.ts` — internal-only `next` param.
- `app/login/page.tsx`, `app/login/LoginClient.tsx`
- `app/signup/page.tsx`, `app/signup/SignupClient.tsx`
- `app/auth/callback/route.ts` — PKCE / email link `exchangeCodeForSession`.

## Implementation Notes (key logic explained)

- **Cookies:** `@supabase/ssr` aligns browser auth with HTTP-only cookies the API layer can read (same pattern Supabase documents for Next.js).
- **`safeInternalPath`:** Rejects `//` and embedded `:` / `\` to avoid open redirects.
- **Signup:** If Supabase returns a session immediately (confirm email disabled), redirect to `next`; else show “check your email” and use `emailRedirectTo` → `/auth/callback?next=…`.

## New env vars (required for auth UI)

- `NEXT_PUBLIC_SUPABASE_URL` — project URL (same host as dashboard API URL).
- `NEXT_PUBLIC_SUPABASE_ANON_KEY` — anon public key.

**Simpler alternative:** Hosted Supabase Auth UI only — not implemented; would still need redirect handling.

## Supabase dashboard checklist

1. Authentication → URL configuration: add `http://localhost:3000/auth/callback` (and production URL) to **Redirect URLs**.
2. Enable **Email** provider.
3. If using email confirmation, keep **Confirm email** on; users complete via link to `/auth/callback`.

## Failure Modes

- Missing public env vars → client throws when submitting forms.
- Redirect URL not allowlisted → email confirm / OAuth fails at Supabase.

## How to Test

1. Add `NEXT_PUBLIC_*` vars to `.env`, restart `npm run dev`.
2. Visit `/signup?next=%2Fbetting%3Fonboard%3D1` — expect 200, not 404.
3. Sign up / sign in → land on `next` path; `/api/user/profile` should return 200 when session exists.

## Verification Checklist

1. `/signup` and `/login` load without 404.
2. After sign-in, `/api/user/profile` is not 401 (same browser).
3. `next=/betting?onboard=1` survives round-trip.
4. `npm run build` succeeds.
