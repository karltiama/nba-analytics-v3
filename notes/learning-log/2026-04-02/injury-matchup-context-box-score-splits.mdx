## Context

We needed a concrete answer to whether “games missed” should come from the injury API history or from box scores, and a first shipped slice of matchup “why” data for Out/Doubtful players.

## Goal

- Define **missed games** using **box-score truth** (no minutes), not injury timeline alone.
- Expose **teammate scoring splits** (with injured player vs without) for key injuries on the betting game details flow.
- Surface **sample-size** caution in UI copy.

## Files Changed

- `lib/betting/injury-matchup-context.ts` — new: queries + `getInjuryMatchupContext(gameId)`.
- `app/api/betting/games/[gameId]/details/route.ts` — attaches `injuryMatchupContext` to JSON.
- `components/betting/MatchupPageLayout.tsx` — types + “When key players didn’t play” section.

## Implementation Notes (key logic explained)

- **Out/Doubtful** players with **season-to-date average minutes (Final games) ≥ 10** (same regex minutes parsing as other betting queries) are candidates — avoids noise from deep bench listings.
- For each candidate, **team Final games** in `analytics.games` for the **same `season`** as the matchup game are split into:
  - **With**: injured player has `player_game_logs` row with parsed minutes **> 0**.
  - **Without**: **no row or minutes ≤ 0** (did not play).
- **Teammates** are the top 12 by average minutes in **with** games; we show up to **4** sorted by largest absolute **PTS delta** (without − with).
- **No new dependencies or env vars.**

## Key Decisions (and alternatives considered)

- **Box scores vs injury history for “missed”:** Injury snapshots describe *status*; they don’t reliably align per-game participation (rest, suspension, DNPs). Box scores encode **who actually played**, which is what we need for splits. Injury history could be layered later as a *label* (“listed out”) but not as the split definition.
- **Threshold 10 mpg:** Tunable constant; lower would add more players (noisier), higher would drop borderline rotation players.
- **Cap 6 injured analyses per request:** Limits worst-case DB load on popular games.

## Assumptions

- `analytics.player_game_logs` and `analytics.games.season` are populated for the current season.
- Players who do not play often have **no row** for that game (treated as missed); if your pipeline always inserts 0-minute rows, those still classify as **without**.

## Failure Modes

- **Trades / two-team rows:** Logs filtered by `team_id`; mid-season trades could mix teams if data is messy — splits are best-effort.
- **Sparse minutes:** Rookies or players with few games yield **low_sample** flag and thin `n` columns.
- **API errors:** Details route catches context errors and still returns injuries list; context falls back to `{ season: '', entries: [] }`.

## How to Test (commands + steps)

1. `npx tsc --noEmit`
2. Run the app, open a betting game page with Out/Doubtful players who have enough game logs (e.g. mid-season).
3. Scroll to **Injuries** — confirm **When key players didn’t play** appears when entries exist.
4. Spot-check one player: manually verify a “without” game is Final and the injured player has no minutes row for that game.
