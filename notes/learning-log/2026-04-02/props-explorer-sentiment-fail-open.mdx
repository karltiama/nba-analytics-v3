## Context
Props were not showing in both the Props Explorer UI and API, likely after sentiment signal enrichment was added to the response pipeline.

## Goal
Make `/api/betting/props-explorer` return rows even if sentiment enrichment fails, instead of failing the whole endpoint.

## Files Changed
- `app/api/betting/props-explorer/route.ts`

## Implementation Notes (key logic explained)
- Wrapped `loadSentimentSignals` in a `try/catch` so sentiment logic becomes best-effort.
- Sanitized `teamIds` by trimming and filtering blank/null values before `ANY($1)` SQL usage.
- Included `r.team_id` in team ID collection to improve signal lookup coverage.
- Guarded per-row processing when `playerTeamId` is missing to avoid bad map lookups.
- On sentiment failure, the function now returns an empty signal map, allowing core props rows to continue.

## Key Decisions (and alternatives considered)
- Decision: fail-open for sentiment only.
  - Why: sentiment is additive metadata; it should not block primary props delivery.
  - Alternative considered: remove sentiment queries entirely. Rejected because it drops useful context.
- Decision: keep all changes inside one function.
  - Why: smallest possible diff and low regression risk.

## Assumptions
- The main outage symptom came from a thrown error inside sentiment SQL/enrichment.
- Returning props without sentiment is acceptable behavior during partial-data issues.

## Failure Modes
- If core props query fails upstream, endpoint still returns 500 (expected).
- If team IDs are malformed in DB rows, sentiment stays empty for those rows.
- If roster/injury tables are temporarily unavailable, sentiment is omitted but rows still return.

## How to Test (commands + steps)
1. Start app and call:
   - `GET /api/betting/props-explorer?date=2026-04-02&limit=50`
2. Confirm response is `200` and `rows.length > 0` when props data exists.
3. Open `/betting/props-explorer` and verify rows render.
4. Verify sentiment badges may appear when available, but absence of sentiment does not hide rows.
5. Check server logs: sentiment failures (if any) should log `props-explorer sentiment fallback` and still return data.
