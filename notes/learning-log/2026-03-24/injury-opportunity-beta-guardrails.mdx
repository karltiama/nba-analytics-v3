## Context
Built an MVP Injury Opportunity Engine (Beta) to detect plausible low-minute opportunity from injury-driven vacated minutes while preventing unrealistic projections.

## Goal
Ship a conservative, explainable first-pass opportunity detector with strict guardrails and a dedicated API output.

## Implementation Notes (key logic explained)
- Added a new service method: `getInjuryOpportunityCandidates(limit)`.
- Candidate pool:
  - upcoming games window
  - low-minute baseline filter (8-22)
  - excludes current Out/Doubtful players
  - requires recent participation
- Injury opportunity pool:
  - computes lost minutes + usage proxy from injured teammates
  - applies starter absorption rule (only 60% redistributed when injured baseline > 30)
- Redistribution:
  - role-aware weighting (`G/F/C/OTHER`)
  - normalized shares, strict pool conservation
- Projection guardrails:
  - `projected_minutes <= 34`
  - `projected_minutes <= baseline + 8`
  - opportunity multiplier clamp `[0.90, 1.15]`
  - anti-double-counting reduction when minute gain is large
- Confidence:
  - penalty-first system for low sample, unstable minutes, weak role match, sparse appearances, uncertain injury context
  - clamp `[0.15, 1.00]`
- Ranking:
  - `edge_vs_line = projected_points - consensus_line_points`
  - `adjusted_edge = edge_vs_line * confidence`
- Added dedicated route:
  - `GET /api/betting/players/opportunity?limit=...`
  - returns engine label, candidates, and metadata.

## Key Decisions (and alternatives considered)
- Used consensus points line from `analytics.player_props_current` (median, min 3 books) for stability.
- Kept role matching simple but strict via position buckets; rejected complex archetype mapping for MVP.
- Tuned game window to 7 days (from 2) after empty-output sanity run, preserving actionable near-term focus.

## Assumptions
- Current injury table is fresh and accurate (`analytics.player_injury_status_current`).
- Enough points over/under lines exist for consensus derivation in active slates.
- Parsed minute values in `analytics.player_game_logs` are sufficiently clean for baseline math.

## Failure Modes
- No nearby games or sparse props can yield zero candidates.
- Position labels may be noisy, which weakens role-match quality.
- Doubtful-heavy injury contexts may lower confidence broadly and suppress ranking spread.

## How to Test (commands + steps)
1. Start app: `npm run dev`
2. Call endpoint:
   - `GET /api/betting/players/opportunity`
   - `GET /api/betting/players/opportunity?limit=10`
3. Validate output fields:
   - projected minutes/points, consensus line, edge, confidence, adjusted edge
   - guardrail flags + injury context
4. Check guardrails:
   - no candidate exceeds `34` minutes
   - no candidate exceeds `baseline + 8`
   - multiplier always between `0.90` and `1.15`
5. Spot-check top candidates for plausible role fit and injury rationale.
