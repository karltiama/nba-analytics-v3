## Context
Injury ingestion was appending raw snapshots and upserting current rows, but stale current injuries were not being removed when players disappeared from new provider pulls.

## Goal
Implement injury sync parity so `analytics.player_injury_status_current` reflects the latest provider state (add/update/remove), and add verification + documentation updates.

## Files Changed
- `lambda/injuries-snapshot/index.ts`
- `scripts/verify-injury-tables.ts`
- `lambda/injuries-snapshot/README.md`
- `notes/learning-log/2026-03-24/injury-sync-parity-step-1-lambda-cleanup.mdx`
- `notes/learning-log/2026-03-24/injury-sync-parity-step-2-verification-check.mdx`
- `notes/learning-log/2026-03-24/injury-sync-parity-step-3-readme-docs.mdx`

## Implementation Notes (key logic explained)
- Lambda transform now tracks player IDs seen in the latest pull and removes current rows for tracked players not in that set.
- Transform result now reports `removed` count for observability.
- Verification script now checks for potential stale rows by comparing current table to latest successful raw pull.
- README now states strict mirror semantics and includes scheduling caveat.

## Key Decisions (and alternatives considered)
- Decision: delete-on-missing in current table to avoid stale injury records impacting APIs.
- Alternative considered: marking a cleared status in-place, but that still keeps stale/current ambiguity in the primary current table.

## Assumptions
- Current table is intended to represent only active injuries.
- Latest successful pull is the canonical comparison point for parity checks.

## Failure Modes
- Provider temporary omissions can produce transient under-reporting due to delete-on-missing behavior.
- If pull frequency is too low, temporary omissions remain visible longer.

## How to Test (commands + steps)
1. Run lambda locally:
   - `cd lambda/injuries-snapshot`
   - `npx tsx index.ts`
2. Confirm log includes `removed` in transform output.
3. Run verification:
   - `npx tsx scripts/verify-injury-tables.ts`
4. Confirm stale-row count is `0` or inspect sample if non-zero.
5. Spot-check injury consumers (game details/matchup) for expected current rows.

## Next Steps
- Optionally add alerting threshold for unusual `removed` spikes.
- Consider a grace-window strategy if provider feed intermittency becomes common.
