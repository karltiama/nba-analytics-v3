## Context
Layer B (`edge_vs_line`, `adjusted_edge`) depends on consensus points lines from `analytics.player_props_current`. We tightened read-time logic only so raw/history ingestion stays unchanged.

## Goal
Improve consensus quality: freshness, latest-per-book snapshots, paired over/under per book, diagnostics on each candidate, and a near-tip stale-line confidence penalty.

## Files Changed
- `lib/betting/queries.ts`

## Implementation Notes (key logic explained)
- **Freshness:** Only rows with `snapshot_at >= now() - (120 * interval '1 minute')` enter consensus (constant `INJURY_OPPORTUNITY_LINE_FRESHNESS_MINUTES`).
- **Latest per book:** `DISTINCT ON (game_id, player_id, sportsbook)` on fresh rows, ordered by `snapshot_at DESC`, `side = over`.
- **Paired O/U:** A book counts only if the same `(game, player, sportsbook, line_value)` has both a fresh over and a fresh under row.
- **Consensus:** Median line (`PERCENTILE_CONT(0.5)`) across qualifying books; `HAVING COUNT(*) >= 3`.
- **Diagnostics:** Each candidate includes `consensus_diagnostics`: `books_count`, `oldest_snapshot_at`, `newest_snapshot_at`, `staleness_minutes` (from DB `now() - newest`), `line_dispersion_stddev` (`stddev_samp` across book lines).
- **Tipoff guard:** If minutes until tip is in `(0, 45)` and `staleness_minutes > 30`, subtract `0.1` from confidence and set `guardrail_flags.near_tip_stale_line_penalty`.
- **Game start:** Upcoming games query now selects `start_time` for the tipoff check.

## Key Decisions (and alternatives considered)
- **120-minute freshness:** Balances injury-news windows vs empty results; tunable via constant.
- **Paired books required:** Stricter than over-only; reduces consensus built from one-sided stale quotes.
- **No DB writes:** All changes are query-side; historical snapshots in raw tables are untouched.

## Assumptions
- `analytics.player_props_current.sportsbook` is populated and comparable across rows.
- `start_time` on `analytics.games` is valid for tipoff proximity.

## Failure Modes
- Stricter filters can reduce or zero out candidates when props are stale or books lack paired lines.
- `stddev_samp` with exactly three points can still be noisy; diagnostics are explanatory, not a hard filter.

## How to Test (commands + steps)
1. `npm run dev`
2. `GET /api/betting/players/opportunity?limit=25`
3. For any candidate, confirm `consensus_diagnostics` is present and timestamps are ISO strings.
4. Confirm `staleness_minutes` is non-negative and plausible vs `newest_snapshot_at`.
5. Manually verify `near_tip_stale_line_penalty` only when within 45m of tip and stale > 30m.
