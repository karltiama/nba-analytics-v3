## Context

The project has four Lambda functions under `lambda/` (nightly-bdl-updater, odds-pre-game-snapshot, boxscore-scraper, player-props-snapshot), deployed manually via AWS CLI. There was no infrastructure-as-code; the goal was to introduce Terraform for one Lambda as a learning step and a base for future expansion.

## Goal

Set up a minimal Terraform layout under `infra/` to manage the existing **nightly-bdl-updater** Lambda: package from current source, IAM role with CloudWatch logs, optional EventBridge schedule, and clear variables/outputs so it’s easy to run and extend.

## Files Changed

- **infra/main.tf** – Terraform and provider version constraints (aws, archive).
- **infra/provider.tf** – AWS provider with region from variable.
- **infra/variables.tf** – `aws_region`, `lambda_function_name`, `lambda_timeout`, `lambda_memory_size`, `lambda_env` (later: `sensitive = true`), `enable_schedule`, `schedule_cron`.
- **infra/outputs.tf** – Lambda name/ARN and optional EventBridge rule name/ARN.
- **infra/iam.tf** – IAM role for Lambda execution, attachment of `AWSLambdaBasicExecutionRole`.
- **infra/lambda.tf** – `archive_file` (zip of `lambda/nightly-bdl-updater`), `aws_lambda_function`, optional EventBridge rule, target, and Lambda permission.
- **infra/terraform.tfvars.example** – Example variables (placeholders for secrets).
- **infra/terraform.tfvars** – Created from example (gitignored); user fills real secrets.
- **infra/.gitignore** – Terraform and zip artifacts.
- **infra/README.md** – How to build Lambda, init/plan/apply, enable schedule, extend to more Lambdas.
- **lambda/nightly-bdl-updater/index.ts** – Only call `pool.end()` when `require.main === module` (local runs), not in Lambda, so warm invocations don’t hit “pool after end”.

## Implementation Notes (key logic explained)

- **Packaging:** `archive_file` uses `source_dir = path.module/../lambda/nightly-bdl-updater` and writes the zip under `infra/` so no extra directory creation is needed. The zip includes `dist/`, `node_modules/`, and `package.json`; the handler is `dist/index.handler` (compiled CommonJS).
- **Build:** Not automated in Terraform. User runs `npm install` and `npm run build` in `lambda/nightly-bdl-updater` before `terraform apply`; this is documented in the README.
- **IAM:** Single role with `lambda.amazonaws.com` assume policy and `AWSLambdaBasicExecutionRole` for CloudWatch logs. No VPC (Supabase is public).
- **EventBridge:** Gated by `enable_schedule` (default false). When true, one `aws_cloudwatch_event_rule`, one `aws_cloudwatch_event_target`, and one `aws_lambda_permission` for `events.amazonaws.com` so the rule can invoke the function.

## Key Decisions (and alternatives considered)

- **Which Lambda first:** Chose nightly-bdl-updater (stats, clear env vars, already documented). Same pattern can be reused for odds, boxscore, player-props, injuries.
- **Zip output path:** Put zip in `infra/` (e.g. `nightly-bdl-updater.zip`) instead of `infra/build/` to avoid creating the parent directory (archive_file does not create it).
- **No null_resource build:** Kept first version simple: no `local-exec` to run npm; user builds manually. Can add a null_resource later if desired.
- **Secrets:** Env vars via `lambda_env` in tfvars or `TF_VAR_*`; no Secrets Manager in v1 to keep scope small.

## Assumptions

- AWS credentials are configured (e.g. AWS CLI or env vars).
- User runs from repo root and builds the Lambda before apply.
- Local state is acceptable; remote backend can be added later.

## Failure Modes

- **Missing or stale build:** If `dist/` is missing or old, the deployed function may fail at runtime. Mitigation: README instructs to run `npm install` and `npm run build` in the Lambda dir before apply.
- **Wrong handler:** If the zip layout or build output changed, handler would need to stay `dist/index.handler` (or match the actual file/export).
- **Pool closed on warm start:** If the Lambda handler calls `pool.end()` in a `finally` block, the first invocation closes the pool; a warm reuse then fails with “Cannot use a pool after calling end on the pool”. Mitigation: only call `pool.end()` when running locally (`require.main === module`).

## How to Test (commands + steps)

1. From repo root: `cd lambda/nightly-bdl-updater`, run `npm install`, `npm run build`, then `cd ../../infra`.
2. Copy `terraform.tfvars.example` to `terraform.tfvars` and set real `lambda_env` (e.g. `SUPABASE_DB_URL`, `BALLDONTLIE_API_KEY`). Confirm `terraform.tfvars` is gitignored (`git check-ignore -v infra/terraform.tfvars`).
3. Install Terraform if needed (e.g. `winget install Hashicorp.Terraform`; Chocolatey was not present). Refresh PATH or open a new shell.
4. **Windows:** Use `-chdir` and full path to var file to avoid “Too many command line arguments”:  
   `terraform -chdir=c:\...\infra plan -var-file=c:\...\infra\terraform.tfvars`  
   Same for `apply -auto-approve`.
5. Mark `lambda_env` as `sensitive = true` in `variables.tf` so plan/apply don’t print secrets.
6. Invoke: `aws lambda invoke --function-name nightly-bdl-updater --payload "{}" infra/response.json` then read `response.json`. Expect `statusCode` 200 and `status: "success"` in body when DB URL and code are correct.

## Verification checklist (done same day)

1. **Build Lambda** – `npm install` and `npm run build` in `lambda/nightly-bdl-updater` (done).
2. **Terraform** – Installed via winget; `terraform init` and `terraform validate` (done).
3. **Variables** – `terraform.tfvars` created, real secrets set, confirmed gitignored (done).
4. **Plan** – Used `-chdir` + full `-var-file` path; plan showed 3 to add (IAM role, attachment, Lambda) (done).
5. **Sensitive** – Added `sensitive = true` to `lambda_env` in `variables.tf` (done).
6. **Apply** – Resources created; Lambda deployed (done).
7. **Invoke** – First: ENOTFOUND (wrong Supabase host; user fixed URL). Second: “pool after end” (handler called `pool.end()`). Fixed by only calling `pool.end()` when `require.main === module`; rebuilt, reapplied, third invoke: success (status 200, pipeline summary returned).
