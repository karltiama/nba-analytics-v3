# Next.js 16 params/searchParams Promise unwrap

## Context

Next.js 16 (and 15+) made `params` and `searchParams` passed to pages/layouts **asynchronous** (Promises). Accessing their properties or enumerating them (e.g. `Object.keys(params)`) before unwrapping triggers runtime errors:

- "params are being enumerated. `params` is a Promise and must be unwrapped with `React.use()` before accessing its properties."
- "The keys of `searchParams` were accessed directly. `searchParams` is a Promise and must be unwrapped with `React.use()` before accessing its properties."

Errors were appearing in the console (often when moving the mouse or clicking), coming from `getReactComponentInfo` / `serializeValue` in the dev overlay when it inspects component props.

## Goal

- Stop console errors about params/searchParams being enumerated.
- Comply with Next.js 16 async dynamic APIs: unwrap params/searchParams before any use.

## Files Changed

- `app/betting/page.tsx` – client page: accept `params`/`searchParams` props and unwrap with `React.use()` at top of component.

## Implementation Notes (key logic explained)

- **Server components (async pages):** Already correct. They use `const { x } = await params` and `await searchParams` at the top; no code change needed.
- **Client components (e.g. betting page):** Next.js still passes `params` and `searchParams` as props (as Promises). If the component doesn’t unwrap them, the dev overlay can enumerate those props when serializing the tree and trigger the error. Fix: accept `params`/`searchParams` in props and call `React.use(props.params)` and `React.use(props.searchParams)` at the very top so the Promises are consumed and not left to be enumerated. The betting page continues to use `useSearchParams()` for actual URL reading; the `use()` calls are only to satisfy the async API and avoid enumeration.

## Key Decisions (and alternatives considered)

- **Unwrap in client page with `use()`:** Chosen so the page doesn’t “hold” raw Promise props that the overlay might serialize. Alternative: ignore and wait for a Next.js fix; we fixed the one client page we control.
- **Leave server pages as-is:** They already await at the top. The overlay may still enumerate the props object Next.js passes; that would require a framework change. No refactor of server pages was done.

## Assumptions

- Next.js passes `params` and `searchParams` to all page default exports, including client components.
- Unwrapping with `use()` in the client page is enough to prevent the enumeration warning for that page; server-component warnings may still appear from overlay serialization until Next.js handles Promise props there.

## Failure Modes

- If `params` or `searchParams` are missing (e.g. non-dynamic route), we guard with `if (props.params)` / `if (props.searchParams)` so `use()` is only called when the Promise exists.
- Calling `use()` unconditionally on `undefined` could throw; the guards avoid that.

## How to Test (commands + steps)

1. `npm run dev` (or `pnpm dev` / `yarn dev`).
2. Open `/betting` and use the page (date filter, etc.).
3. Move the mouse over the page and click around; watch the browser console.
4. Visit dynamic routes (e.g. `/players/[playerId]`, `/teams/[teamId]`, `/games/[gameId]`) and repeat; server pages should still work and any remaining warnings may come from the dev overlay, not from direct misuse.

## Next Steps

- If server pages still log params/searchParams enumeration errors, consider opening an issue or checking Next.js canary for a fix in the dev overlay’s serialization of Promise props.
- When adding new client pages that receive `params`/`searchParams`, unwrap them with `React.use()` at the top.
