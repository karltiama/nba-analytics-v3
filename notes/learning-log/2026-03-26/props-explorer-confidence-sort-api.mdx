## Context
Props Explorer already supported server-side computed sorting for EV-based fields, while confidence tier was displayed but not sortable.

## Goal
Add backend support for `sort=confidence` so API responses can be ordered by confidence tier.

## Files Changed
- `app/api/betting/props-explorer/route.ts`

## Implementation Notes (key logic explained)
- Expanded computed-sort detection from EV-only to a new `COMPUTED_SORTS` set that includes `confidence`.
- Extended `sortKey()` with confidence ranking: `high=3`, `medium=2`, `low=1`, missing/unknown -> negative infinity.
- Reused the existing computed-sort pipeline (fresh fetch cap, in-memory sort, pagination slice) for confidence sorting.

## Key Decisions (and alternatives considered)
- Used numeric ranking for confidence to keep ordering deterministic and simple.
- Chose to reuse the existing computed-sort flow instead of adding SQL CASE sorting, because confidence is model-derived and already computed post-query.

## Assumptions
- Confidence tiers are constrained to `high`, `medium`, `low`, or null.
- Default direction for confidence sorting will be controlled by UI/query params; descending maps to strongest confidence first.

## Failure Modes
- Unknown confidence values (if introduced later) will sort as lowest priority.
- Very large slates still rely on fetch-cap behavior, so global ordering remains approximate outside the capped window.

## How to Test (commands + steps)
1. Start app: `npm run dev`
2. Open `/betting/props-explorer?sort=confidence&dir=desc`
3. Confirm rows appear with `high` confidence tiers before `medium` and `low`.
4. Flip to `dir=asc` and verify reverse ordering.
5. Check API directly: `/api/betting/props-explorer?sort=confidence&dir=desc&limit=20`

## Next Steps
- Add `confidence` to the Props Explorer sort dropdown so users can trigger this without URL edits.
