## Context
The research table on `BettingResearchPage` displayed numeric player identifiers (for example, `227`) instead of readable player names for some rows.

## Goal
Ensure the API provides a player name whenever possible so the existing UI cell rendering shows names instead of raw IDs.

## Files Changed
- `app/api/betting/research/prop-eval/route.ts`

## Implementation Notes (key logic explained)
Updated the main row query to:
- alias `research.v_prop_eval_units` as `v`
- `LEFT JOIN analytics.players p` using `player_id` text matching
- return `COALESCE(v.player_name, p.full_name) AS player_name`

This preserves existing behavior when `v.player_name` exists, while filling gaps from the canonical players table when it does not.

## Key Decisions (and alternatives considered)
- Chosen: resolve fallback name in SQL query layer with `COALESCE`.
- Alternative: do an additional lookup pass in application code after query results are loaded.
- Why chosen: single query, simpler data flow, no extra in-memory mapping pass.

## Assumptions
- `analytics.players` exists and contains valid `full_name` values for most `player_id` values present in `research.v_prop_eval_units`.
- `player_id` can be safely compared as text across both sources.

## Failure Modes
- If a `player_id` is missing in `analytics.players`, UI still falls back to showing `playerId`.
- If the `analytics.players` relation is unavailable in an environment, the endpoint query could fail.

## How to Test (commands + steps)
1. Run app: `npm run dev`
2. Open `/betting/research`
3. Confirm rows that previously showed numeric IDs now show names where available.
4. Spot-check one known player id against `analytics.players.full_name`.
5. Verify filters (`after`, `before`, `prop_type`) still return data and sort order is unchanged.
