## Context
The research prop-eval endpoint currently exposes only the last pre-tip snapshot, which hides pregame movement.

## Goal
Add pregame timing buckets (T-60m, T-15m, Close) so downstream API/UI can use movement context.

## Files Changed
- `db/schemas/research_v_prop_decision_lines.sql`
- `db/schemas/research_v_prop_eval_units.sql`
- `db/schemas/research_install_all.sql`

## Implementation Notes (key logic explained)
Extended `research.v_prop_decision_lines` from a close-only view to include bucketed pregame fields:
- `close_*` fields from the existing closing row
- `t60m_*` via lateral subquery at or before `game_start_time - 60 minutes`
- `t15m_*` via lateral subquery at or before `game_start_time - 15 minutes`

Kept original close columns (`odds_american`, `odds_decimal`, `implied_probability`) for compatibility.

## Key Decisions (and alternatives considered)
- Chosen: LATERAL lookups from each close row to find nearest prior bucket snapshots.
- Alternative: Window-function-based bucketing CTE over all snapshots.
- Why chosen: easiest to read and minimal disruption to existing logic.

## Assumptions
- Snapshot cadence is sufficient that many rows have T-60m/T-15m values.
- Matching key `(game_id, player_id, sportsbook, prop_type, side)` is stable across snapshots.

## Failure Modes
- Missing snapshots for a bucket produce `NULL` bucket fields.
- High row volume may increase view compute cost due to lateral lookups.

## How to Test (commands + steps)
1. Apply schema SQL in DB.
2. Query `research.v_prop_decision_lines` for known games.
3. Verify `close_*`, `t60m_*`, `t15m_*` columns populate where expected.
4. Spot-check one market timeline to confirm bucket timing constraints.

## Next Steps
- Project these fields into `research.v_prop_eval_units`.
- Add implied probability deltas for movement analysis.
