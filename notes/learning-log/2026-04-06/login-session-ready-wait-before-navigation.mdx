## Context
In production, users could be redirected back to `/login?next=/betting?onboard=1` after successful credential sign-in, requiring a second click.

## Goal
Prevent first-login redirect loops by ensuring session state is available before navigating to protected routes.

## Files Changed
- `app/login/LoginClient.tsx`

## Implementation Notes (key logic explained)
- Added `waitForSessionReady()` helper:
  - polls `supabase.auth.getSession()` up to 8 attempts
  - waits 120ms between attempts
  - returns once `session.user` exists
- After `signInWithPassword` success, navigation now waits for session readiness before `router.push(nextPath)`.
- This addresses race conditions between auth success and middleware-protected route checks.

## Key Decisions (and alternatives considered)
- Chose a short bounded poll loop as a minimal, explicit fix.
- Alternative considered: reintroducing `router.refresh()`; rejected due extra render/network overhead.

## Assumptions
- The issue is caused by session hydration/cookie propagation lag rather than invalid credentials.

## Failure Modes
- If auth provider/session API is unusually slow, user may still perceive slight delay before navigation.
- If session never materializes due upstream auth error, flow still fails (existing error handling remains).

## How to Test (commands + steps)
1. Open `/login?next=%2Fbetting%3Fonboard%3D1`.
2. Submit valid email/password once.
3. Confirm immediate successful navigation to `/betting?onboard=1` without returning to login.
4. Repeat after idle period to simulate cold state.
