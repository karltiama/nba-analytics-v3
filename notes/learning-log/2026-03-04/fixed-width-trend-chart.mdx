## Context
The PlayerTrendChart on the betting player detail page was using a `ResizeObserver` to
stretch the SVG to fill the container width. With `preserveAspectRatio="none"`, text and
dots distorted as the container resized. The x-axis also grew wider with more data points
due to `PX_PER_POINT` scaling, making the chart sprawl horizontally.

## Goal
Make the chart a fixed width so the x-axis never scales. Instead, the *density* of data
points increases as more games are added. Shrink the overall horizontal footprint so it
flows better in the layout.

## Files Changed
- `app/betting/players/[playerId]/components/PlayerTrendChart.tsx`

## Implementation Notes
- Replaced dynamic width calculation (`ResizeObserver` + `PX_PER_POINT`) with a single
  constant `SVG_WIDTH = 540`.
- The plot area is `540 - 48 (left) - 48 (right) = 444px`. All data points share this
  space equally: `step = 444 / (data.length - 1)`.
- Changed `preserveAspectRatio` from `"none"` to `"xMidYMid meet"` so text, dots, and
  lines scale uniformly if the SVG is rendered smaller than 540px.
- Added `max-w-full` to the SVG so it shrinks on narrow viewports instead of overflowing.
- Removed `useEffect` import since `ResizeObserver` is no longer used.
- Removed `containerRef`, `containerWidth` state, and `overflow-x-auto` from the wrapper div.

## Key Decisions (and alternatives considered)
| Decision | Why | Alternative |
|---|---|---|
| Fixed 540px width | Gives a compact chart that doesn't sprawl; ~60% of a typical content column | Could use a percentage of container width, but user specifically wanted no x-axis scaling |
| `xMidYMid meet` | Preserves aspect ratio and centers the chart | `xMinYMid meet` would left-align — could revisit |
| Removed ResizeObserver entirely | Not needed when width is fixed | Could keep it to clamp SVG to container, but `max-w-full` handles the overflow case |

## Assumptions
- 540px fits comfortably in the player detail page layout (content area is ~1200px).
- Label thinning (`showEveryNthLabel`) is sufficient for dense datasets; no need for a
  scrollable x-axis.
- The hover mouse-position math still works correctly because `scaleX` accounts for
  the ratio between viewBox width and rendered width.

## Failure Modes
- If the chart renders in a very narrow container (<300px), labels may overlap. The
  `showEveryNthLabel` logic mitigates this but doesn't fully prevent it.
- With 50+ data points, dots will be ~9px apart — visually dense but still readable.
  At 80+ points it may be hard to hover individual dots.

## How to Test (commands + steps)
1. `npm run dev` and navigate to a player's betting detail page.
2. Check chart with a player who has <10 games — dots should be generously spaced.
3. Check chart with a player who has 20-30 games — dots should be tighter, same width.
4. Resize the browser window narrow (<600px) — chart should shrink proportionally.
5. Hover over data points — tooltip should still appear with correct values.

## Next Steps
- Consider adding a scroll/pan gesture if the dataset exceeds ~40 points.
- Could animate the dot size smaller as density increases for better UX.
- Evaluate if 540px feels right across different page layouts or if it should be a prop.
