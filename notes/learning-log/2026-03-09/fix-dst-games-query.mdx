## Context

The betting dashboard (`/betting`) was displaying 13 games for "today" instead of 5. The extra 8 games were from the previous day (March 8), all with status "Final".

## Goal

Fix the `getGamesForDate` SQL query so that it correctly filters to only today's games when DST transitions occur.

## Files Changed

- `lib/betting/queries.ts` — primary fix in `getGamesForDate()`
- `app/api/games/route.ts` — same pattern used in the general games API
- `lambda/player-props-snapshot/index.ts` — same pattern in the Lambda function

## Implementation Notes

The fix is a one-word change in each query: `$1::date` → `$1::timestamp` in the `AT TIME ZONE` boundary computation.

**Why this matters:**

PostgreSQL handles `date AT TIME ZONE 'zone'` differently than `timestamp AT TIME ZONE 'zone'`:

1. `date` is implicitly promoted to `timestamptz` using the **session timezone** (UTC on Supabase)
2. `timestamptz AT TIME ZONE 'zone'` returns `timestamp WITHOUT time zone` (wall-clock time in that zone)
3. When this `timestamp without time zone` is compared to a `timestamptz` column, PostgreSQL implicitly converts it back to `timestamptz` using session tz (UTC) — producing the WRONG boundary

With `timestamp` (without time zone):

1. `$1::timestamp` gives `2026-03-09 00:00:00` (no timezone)
2. `timestamp_without_tz AT TIME ZONE 'zone'` returns `timestamptz` — correctly interpreted as midnight in that zone
3. Result: `2026-03-09 04:00:00+00` (midnight EDT = 4am UTC) ✓

## Key Decisions

| Decision | Chosen | Alternative |
|---|---|---|
| Cast approach | `$1::timestamp` | `($1 \|\| ' 00:00:00 America/New_York')::timestamptz` or `make_timestamptz()` |
| Scope | Fix all 3 files with same pattern | Fix only the primary query |

Chose `::timestamp` because it's the minimal change, keeps the query readable, and the `AT TIME ZONE` semantics are correct for `timestamp without time zone` input.

## Assumptions

- Supabase session timezone is UTC (confirmed via `current_setting('TIMEZONE')`)
- All game `start_time` values are stored as `timestamptz` (confirmed via `information_schema.columns`)
- The date string passed to `getGamesForDate` is always in `YYYY-MM-DD` format

## Failure Modes

- If Supabase ever changes the session timezone from UTC, both the old and new queries would need review
- If `start_time` were changed to `timestamp without time zone`, the comparison semantics would change
- This bug only manifests around DST transitions (2x per year), making it easy to miss in testing
