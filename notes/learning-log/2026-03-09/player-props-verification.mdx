## Implementation Notes (key logic explained)

### Bug 1: Column name mismatch — `game_date` does not exist

The `getTodayGameIds()` function queried `analytics.games` using a `game_date` column that doesn't exist. The actual column is `start_time` (timestamptz). Both the primary query and the fallback query were updated:

```sql
-- Before (broken)
WHERE game_date = $1

-- After (working)
WHERE start_time::date = $1::date
```

### Bug 2: `bdl_prop_id` integer overflow

BDL player prop IDs (e.g., `5074535198`) exceed PostgreSQL's `integer` max of `2,147,483,647`. The `bdl_prop_id` column was changed from `integer` to `bigint` in three tables:
- `raw.player_prop_snapshots`
- `analytics.player_prop_current`
- `analytics.player_prop_history`

The live database was also patched with `ALTER TABLE ... ALTER COLUMN bdl_prop_id TYPE bigint`.

### BDL API cross-game response behavior

During verification we discovered that the BDL `/v2/odds/player_props?game_id=X` endpoint sometimes returns props for other games in the same response. This caused the raw layer to store ~10,666 rows instead of the expected 7,204 (from 3 games). The extra rows are props for games 18447750 and 18447751 that appeared in the game 18447749 response.

This is a BDL API behavior, not a code bug. The analytics layer handles it correctly:
- `player_prop_current` uses delete-then-insert per game/vendor, so stale data is replaced
- `player_prop_history` uses `ON CONFLICT DO NOTHING`, so duplicate market states are skipped
- Raw tables are append-only by design, so extra rows are harmless

## Key Decisions (and alternatives considered)

| Decision | Rationale | Alternative |
|---|---|---|
| Patch live DB with `ALTER COLUMN` instead of recreating tables | Tables were just created with 0 rows, ALTER is safe | Could have dropped and recreated, but ALTER is cleaner |
| Keep cross-game raw duplicates | Append-only raw is working as designed; analytics layer dedupes | Could add `WHERE game_id = <requested>` filter in Lambda, but that would discard valid data |
| Use `start_time::date` cast | Clean single-column comparison; `analytics.games` uses `start_time` timestamptz, no separate date column | Could add a `game_date` computed column, but unnecessary complexity |

## Assumptions

- BDL API `bdl_prop_id` values will continue to be > 2B (bigint is safe up to 9.2 quintillion)
- The cross-game API behavior is consistent — the Lambda processes all returned rows regardless of which game was queried
- `analytics.games.start_time` always has the game date in UTC

## Failure Modes

| Failure | Impact | Mitigation |
|---|---|---|
| `start_time` is null for some games | `start_time::date` comparison would fail to match | Fallback query returns all games for the date regardless of status |
| BDL changes prop ID format | If IDs become UUIDs or strings, `bigint` column would break | `bdl_prop_id` is nullable, so the row would still insert; Zod schema would need updating |
| Individual INSERT performance | ~12 min for 7K props is slow for Lambda (15 min timeout) | Future optimization: batch INSERTs using multi-value `VALUES` clauses |
