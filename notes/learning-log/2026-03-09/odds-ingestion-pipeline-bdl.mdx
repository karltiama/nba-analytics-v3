## Context

The app already ingests NBA stats/games from BallDontLie (BDL) and Basketball Reference into `raw.*` and `analytics.*` schemas. Odds were previously fetched from The Odds API and stored in `public.markets` (upserted, no history preserved). The goal was to build a proper odds ingestion pipeline with historical snapshots using BDL's `/v2/odds` endpoint.

## Goal

Implement an odds ingestion workflow that:
1. Pulls NBA game odds from BallDontLie `/v2/odds` (GOAT tier)
2. Stores raw odds snapshots in append-only `raw.*` tables
3. Transforms them into clean `analytics.*` tables
4. Supports current game card display and future line movement analysis

## Implementation Notes

### Data Source: BDL `/v2/odds`

The BDL odds endpoint returns flat rows â€” one per game+vendor â€” with all three markets (spread, moneyline, total) inline. This is much simpler than The Odds API which returns nested bookmakers/markets/outcomes.

Key advantage: BDL returns `game_id` (integer) that maps directly to our canonical `games.game_id` via `String(bdl_game_id)`. No team-name matching or schedule lookup needed.

### Raw Schema (2 tables)

- `raw.odds_pull_runs`: one row per Lambda invocation, tracks status/timing/metadata
- `raw.odds_snapshots`: one row per BDL response row (game+vendor), mirrors API shape exactly

Both are append-only â€” no unique constraints, no updates, no deletes.

### Analytics Schema (3 tables)

- `analytics.game_odds_current`: upserted, one row per game with latest odds from preferred vendor
- `analytics.game_odds_history`: append-only timeline, unique on `(game_id, vendor, snapshot_at)` to prevent dupes on re-runs
- `analytics.game_line_movement_summary`: convenience table computed from first/last history snapshots

### Transform Flow

The Lambda runs the transform inline after storing raw data. The standalone script exists for re-runs and backfills. Both use the same logic:
1. UPSERT current odds from latest snapshot per game (preferred vendor)
2. INSERT history rows with ON CONFLICT DO NOTHING
3. Refresh line movement summary using CROSS JOIN LATERAL for first/last snapshots

### Query Layer

`getGameOdds`/`getGamesOdds` now read from `analytics.game_odds_current` first (single flat row per game â€” no aggregation needed). Falls back to `public.markets` for games not yet in analytics. The `GameOdds` interface is unchanged so frontend components need zero changes.

## Key Decisions (and alternatives considered)

| Decision | Why | Alternative |
|---|---|---|
| BDL `/v2/odds` over The Odds API | Direct game_id mapping, same API key, flat response | The Odds API: cheaper but requires team-name matching |
| 2 raw tables instead of 4 | BDL response is already flat (one row = all markets) | Normalize into events/markets/outcomes â€” unnecessary complexity |
| Inline transform in Lambda | Avoids a separate scheduled step for analytics freshness | Separate transform Lambda/cron â€” adds deployment complexity |
| Legacy fallback in query layer | Preserves existing `public.markets` data during transition | Hard cutover â€” risks breaking frontend if analytics tables empty |

## Assumptions

- `BALLDONTLIE_API_KEY` has GOAT tier access ($39.99/mo required for `/v2/odds`)
- Games are seeded in `analytics.games` before odds ingestion runs (via nightly BDL updater)
- DraftKings is the preferred vendor for the single-vendor current odds view
- BDL `game_id` (integer) maps to our `games.game_id` via `String()` conversion

## Failure Modes

| Scenario | Handling |
|---|---|
| BDL API returns 429 (rate limit) | Lambda retries after 60s sleep |
| BDL API returns non-200 | Pull run marked as 'error', error message stored |
| Game not in `analytics.games` | Raw snapshot stored (game_id preserved), analytics transform skips it |
| Duplicate pull for same time | Raw rows appended (append-only), analytics history deduped by unique constraint |
| Transform fails mid-run | Pull run stays 'started', re-running script will re-process |
| `analytics.game_odds_current` empty | Query layer falls back to `public.markets` |

## How to Test

### 1. Apply migrations

```bash
# Run raw odds schema
psql $SUPABASE_DB_URL -f db/schemas/raw_odds_schema.sql

# Run analytics odds schema
psql $SUPABASE_DB_URL -f db/schemas/analytics_odds_schema.sql
```

### 2. Run Lambda locally

```bash
# Today's games
npx tsx lambda/odds-pre-game-snapshot/index.ts

# Specific date
npx tsx lambda/odds-pre-game-snapshot/index.ts 2026-03-09
```

### 3. Verify raw data

```sql
SELECT * FROM raw.odds_pull_runs ORDER BY pull_run_id DESC LIMIT 5;
SELECT count(*), count(DISTINCT game_id), count(DISTINCT vendor) FROM raw.odds_snapshots;
```

### 4. Verify analytics data

```sql
SELECT * FROM analytics.game_odds_current LIMIT 5;
SELECT count(*) FROM analytics.game_odds_history;
SELECT * FROM analytics.game_line_movement_summary LIMIT 5;
```

### 5. Run standalone transform (re-run test)

```bash
npx tsx scripts/transform-raw-odds-to-analytics.ts
npx tsx scripts/transform-raw-odds-to-analytics.ts --vendor fanduel
```

### 6. Verify frontend

Open `/betting` in browser â€” game cards should display odds from `analytics.game_odds_current`.
