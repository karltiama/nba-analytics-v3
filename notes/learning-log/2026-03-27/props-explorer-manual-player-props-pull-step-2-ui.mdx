## Context
After adding a backend manual trigger for the player-props Lambda, the Props Explorer page still needed a direct control to run it without leaving the UI.

## Goal
Add a single UI action that invokes the manual Lambda trigger and refreshes the current props table.

## Files Changed
- `app/betting/props-explorer/page.tsx`
- `notes/learning-log/2026-03-27/props-explorer-manual-player-props-pull-step-2-ui.mdx`

## Implementation Notes (key logic explained)
- Added `manualPulling` state to track button loading/disabled state.
- Added `handleManualPull` callback that:
  - `POST`s to `/api/betting/props-explorer`
  - on success, reloads rows using existing active filters/sort/pagination
  - surfaces any failure through existing `error` UI state
- Added a `Pull Player Props` button in the filters row:
  - disabled while manual pull or page load is in progress
  - label changes to `Pulling props...` while running

## Key Decisions (and alternatives considered)
- Kept logic local to this page to avoid introducing shared abstractions in this change.
- Reused existing error display instead of adding new toast/notification system.
- Alternative considered: optimistic success message and delayed polling loop; skipped to keep diff small.

## Assumptions
- Lambda execution time is usually short enough that one immediate reload is useful.
- If ingestion takes longer, user can click the button again or adjust filters/date.

## Failure Modes
- Trigger endpoint returns non-200 due to AWS auth/function issues.
- Trigger succeeds but data is not visible yet due to ingestion lag.
- Network errors during trigger or follow-up reload.

## How to Test (commands + steps)
1. Run app: `npm run dev`
2. Open: `http://localhost:3000/betting/props-explorer`
3. Click `Pull Player Props`.
4. Verify button shows `Pulling props...` and is disabled while running.
5. Verify table refreshes after request completes.
6. If trigger fails, verify red error card appears with message.
