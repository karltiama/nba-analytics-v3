## Context
Manual pull controls were added to Props Explorer, but the requirement changed: end users should not be able to trigger Lambda execution from the UI.

## Goal
Remove the user-facing manual pull action from the Props Explorer page.

## Files Changed
- `app/betting/props-explorer/page.tsx`
- `notes/learning-log/2026-03-27/props-explorer-disable-manual-pull-step-1-ui.mdx`

## Implementation Notes (key logic explained)
- Removed `manualPulling` state and all callback logic related to manual trigger.
- Removed local helper used to re-fetch rows after manual trigger.
- Removed `Pull Player Props` button from the filter controls.
- Left existing read-only filtering and data-loading flow unchanged.

## Key Decisions (and alternatives considered)
- Full removal chosen over role-gated rendering to enforce policy simply.
- Kept API-side investigation separate so this UI diff is minimal and easy to review.

## Assumptions
- Users should only consume pulled data and not initiate ingestion from web UI.

## Failure Modes
- None introduced functionally; this only removes a control.
- If manual trigger remains on backend, direct API calls could still invoke it (addressed in next step).

## How to Test (commands + steps)
1. Run app: `npm run dev`
2. Open `http://localhost:3000/betting/props-explorer`
3. Confirm there is no `Pull Player Props` button.
4. Verify filters, sorting, and table loading still work.

## Next Steps
- Remove the backend `POST /api/betting/props-explorer` trigger route to fully disable manual invocation.
- Continue investigation of scheduled ingestion behavior and timing windows.
