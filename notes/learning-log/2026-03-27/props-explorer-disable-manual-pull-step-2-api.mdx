## Context
After removing manual pull from UI, backend still exposed a `POST` route that could invoke Lambda directly.

## Goal
Remove backend manual Lambda invocation from Props Explorer API route.

## Files Changed
- `app/api/betting/props-explorer/route.ts`
- `notes/learning-log/2026-03-27/props-explorer-disable-manual-pull-step-2-api.mdx`

## Implementation Notes (key logic explained)
- Removed Node process invocation imports (`execFile`, `promisify`).
- Removed Lambda function candidate constants and AWS region constant.
- Removed `invokePlayerPropsLambda()` helper.
- Removed `POST /api/betting/props-explorer` handler entirely.
- `GET /api/betting/props-explorer` remains unchanged for read-only explorer data.

## Key Decisions (and alternatives considered)
- Hard removal chosen over auth-gating to eliminate accidental or unauthorized invocation paths.
- Keeping route read-only simplifies operational posture and frontend behavior.

## Assumptions
- Manual ingestion control should be done outside user-facing app routes (e.g., ops scripts, AWS console, scheduler).

## Failure Modes
- None for read-only behavior.
- Any existing automation relying on the removed `POST` endpoint will now fail (intentional policy change).

## How to Test (commands + steps)
1. Start app: `npm run dev`
2. Verify read path works:
   - `GET /api/betting/props-explorer?...` returns rows/meta.
3. Verify manual invoke path is disabled:
   - `POST /api/betting/props-explorer` should return 405 (Method Not Allowed).

- Optionally add admin-only observability endpoint for latest pull status (read-only, no trigger).
